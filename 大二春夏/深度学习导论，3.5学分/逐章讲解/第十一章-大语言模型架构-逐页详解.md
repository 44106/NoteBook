## 第 3 页 · 模型基础（一）——Transformer 的基石作用

**PPT 内容**：
- Transformer 灵活的并行架构为训练数据和参数的扩展提供了模型基础，推动了本轮大语言模型的发展。

### 逐页讲解

这一页重申了一个关键事实：**当前所有大语言模型，无论是 GPT-4、Claude、DeepSeek、LLaMA，全部建立在 Transformer 架构之上。** 没有任何一个主流大语言模型采用了其他架构（如纯 RNN、纯 CNN、或者某种全新的结构）。

为什么 Transformer 能成为这个唯一的基座？两个核心原因：

第一，**自注意力机制的并行性**。RNN 必须串行处理——读完第 t 个 Token 才能处理第 t+1 个。但 Transformer 的自注意力机制让所有 Token 可以同时处理——每个 Token 通过 Q（Query）、K（Key）、V（Value）矩阵并行的方式与所有其他 Token 交互。这充分利用了 GPU 的大规模并行计算能力——一块 H100 GPU 有几万个计算核心，Transformer 能让它们同时工作，而 RNN 只能让极少部分核心在任一时刻处于活跃状态。

第二，**自注意力机制的长程建模能力**。RNN 中相距较远的两个 Token 之间的信息传递需要经过多个时间步的循环——每一步都可能丢失一部分信息。但在 Transformer 的自注意力中，任意两个 Token——无论它们在序列中相隔多远——在每一层都有一次**直接的交互**（距离恒为 1，只要 attention score > 0）。这让模型可以轻松捕捉跨段落、跨章节的长程语义依赖——这是理解复杂文本和进行逻辑推理的基础。

这两个特性——**并行性**和**长程建模**——为「规模化」扫除了技术障碍。如果没有 Transformer，我们不可能训练千亿甚至万亿参数级别的模型——RNN 的串行瓶颈会让训练时间长得不切实际。


## 第 4 页 · 模型基础（二）——Transformer 的扩展能力

**PPT 内容**：
- 继续第 3 页的论述——Transformer 灵活的并行架构为训练数据和参数的扩展提供了模型基础，推动了本轮大语言模型的发展。

### 逐页讲解

这一页是对第 3 页的延伸和强化。Transformer 架构的「可扩展性」体现在多个维度：

**深度扩展**：Transformer 的残差连接（Residual Connection）+ 层正则化（Layer Normalization）机制让它可以堆叠非常深——GPT-3 有 96 层，LLaMA-3.1 405B 有 126 层——而不会出现梯度消失或训练不稳定的问题。ResNet 在计算机视觉中验证的「残差连接可以让网络变得更深」的结论，在 Transformer 中同样成立。

**宽度扩展**：注意力头的数量（从 12 到 128 甚至更多）、隐藏层维度（从 768 到 12288 甚至更高）、FFN 中间层维度都可以线性扩展。而且这些扩展带来的计算量增长是可预测的——你可以很精确地估算扩大模型需要多少算力。

**数据扩展**：Transformer 的并行训练让它可以高效处理海量数据。GPT-3 的训练数据量约 1TB（~300B Token），LLaMA-3 高达约 50TB（~15T Token）。如果使用 RNN，在同样的时间内只能处理这些数据量的极小一部分。

**分布式扩展**：Transformer 的矩阵乘法为主的运算模式非常适合模型并行（把模型切分到多块 GPU 上）、数据并行（每块 GPU 处理不同的数据批次）、流水线并行（不同层放在不同 GPU）等多种分布式训练策略的组合。这就是为什么我们可以用几千甚至上万块 GPU 同时训练一个模型。

正因为 Transformer 在所有这些维度上都展现出了极佳的可扩展性，它才成为了大语言模型时代的「唯一基座」。它就像乐高积木——你可以用同样的基本单元，搭出从小型到超大型的各种规模的模型，而架构的本质不需要做任何改变。


## 第 5 页 · 三种架构总览

**PPT 内容**：
- 在 Transformer 的基础上衍生出了三种主流模型架构：
- Encoder-only 架构——只选用 Transformer 中的 Encoder 部分，代表模型为 BERT、RoBERTa 等
- Encoder-Decoder 架构——同时选用 Transformer 中的 Encoder 和 Decoder 部分，代表模型为 T5、BART 等
- Decoder-only 架构——只选用 Transformer 中的 Decoder 部分，代表模型为 GPT 系列、LLaMA 系列等

### 逐页讲解

⭐ 这一页是全章最重要的概念分岔点。Transformer 的原始论文（「Attention Is All You Need」, 2017）提出的是一个完整的 Encoder-Decoder 架构——用于机器翻译任务。Encoder 负责「读懂」源语言句子，Decoder 负责「写出」目标语言句子。但在后续的发展中，研究者发现这个完整架构可以被「拆开」使用，由此衍生出了三种不同的路线。

三者的本质区别在于**注意力矩阵的掩码形状**：

**Encoder-only（完全注意力 / 双向注意力）**：在 Encoder 的自注意力层中，每个 Token 能看到序列中的**所有**其他 Token——不仅能看到它前面的词（上文），也能看到它后面的词（下文）。这使得 Encoder 能够「双向」地理解每个 Token 在完整上下文中的语义——非常适合**理解类任务**（文本分类、情感分析、命名实体识别、语义相似度计算）。但 Encoder 没有「生成」能力——它不能逐个 Token 地输出文本序列。

**Decoder-only（下三角因果掩码 / 单向注意力）**：在 Decoder 的自注意力层中，每个 Token **只能看到它自己以及之前的 Token**，不能看到未来的 Token。这是为了保证自回归生成的因果一致性——你在生成第 t 个 Token 时，不能「偷看」第 t+1 个 Token。这种架构天然适合**生成类任务**（文本续写、对话、代码生成）。因为不需要 Encoder 组件——参数量更集中、架构更简洁、可扩展性更强。

**Encoder-Decoder（双向编码 + 因果解码 + 交叉注意力）**：同时包含 Encoder 和 Decoder。Encoder 用双向注意力理解输入序列；Decoder 用因果注意力逐 Token 生成输出序列；Decoder 的每一层还通过**交叉注意力（Cross-Attention）**来引用 Encoder 的完整输出。这使得模型既能有深度的输入理解，又能生成输出——非常适合需要「输入和输出在语义上有强对应关系」的任务（机器翻译、文本摘要）。但代价是参数量翻倍（Encoder 和 Decoder 各占一半左右），训练和推理成本更高。


## 第 6 页 · Encoder-only 架构详解

**PPT 内容**：
- Encoder-only 架构只选取了 Transformer 中的编码器部分，包含三个部分：输入编码部分（将输入文本转化为向量序列）、特征编码部分（为向量序列进一步提取和深化文本特征）、任务处理部分（根据任务需求生成输出）。

### 逐页讲解

这一页深入到 Encoder-only 架构的内部结构，将其拆分为三个阶段：

**阶段一：输入编码。** 原始文本首先经过分词（Tokenization），变成 Token ID 序列。每个 Token ID 通过词嵌入矩阵（Embedding Matrix）映射为一个稠密向量（比如 768 维），再叠加上位置编码（Positional Encoding）以注入序列顺序的信息。这一步的输出是一个「向量序列」——每个 Token 对应一个语义向量，但这个向量还非常原始，只包含了这个词的基本语义和位置信息，缺乏上下文的交互。

**阶段二：特征编码。** 这是 Encoder 的核心——由多个完全相同的 Encoder Block 堆叠而成（BERT-Base 有 12 层，BERT-Large 有 24 层）。每一层包含一个多头自注意力子层（Multi-Head Self-Attention）+ 一个全连接前馈子层（FFN），每个子层后面都有残差连接（Residual Connection）和层正则化（Layer Normalization）。经过多层的逐级处理，每个 Token 的向量从最初孤立的词嵌入逐渐演化为融合了整个句子上下文信息的「语境化表征」。第 1 层的注意力可能只捕捉到相邻词的局部搭配关系；第 12 层的注意力可能已经捕捉到了跨句子的语义关联。

**阶段三：任务处理。** Encoder-only 模型不能做「生成」——它没有 Decoder。所以它需要针对不同的下游任务设计不同的「输出头」。最常见的做法是取第一个 Token（[CLS] Token）在最后一层的输出向量——这个向量经过所有层的自注意力已经聚合了整个输入序列的全局语义——然后将它送入一个简单的线性分类器（比如一个全连接层 + Softmax），输出分类结果。对于 Token 级别的任务（比如命名实体识别），则对每个 Token 的输出向量分别做分类。

三个阶段的设计非常清晰：通用编码 → 逐层深化 → 特定输出。这种模块化使得 Encoder-only 模型在各类**理解型 NLP 任务**（情感分析、文本分类、句子相似度判断、问答系统中的阅读理解）上表现极其出色。


## 第 7 页 · Decoder-only 架构详解

**PPT 内容**：
- Decoder-only 架构选取了 Transformer 中的解码器部分，即输入编码、特征解码及输出生成部分。
- 核心特点在于**省略了每个解码模块中的交叉注意力子模块**（因为不再有独立的 Encoder 供交叉注意力去引用）。
- 处理流程：将输入文本转化为带位置信息的向量序列 → 处理向量序列并自回归生成输出序列 → 将解码后的向量转换为概率分布并采样 Token 输出。

### 逐页讲解

Decoder-only 架构的最大特点是**极简**。标准的 Transformer Decoder 原本有三个子层——自注意力（带因果掩码）、交叉注意力（引用 Encoder 的输出）、前馈网络。但 Decoder-only 架构中去掉了 Encoder，因此也去掉了交叉注意力子层——只剩下两个子层：**因果自注意力 + 前馈网络**。

处理流程拆解为三步：

**第一步：输入编码。** 和 Encoder-only 类似——输入 Token 经嵌入矩阵映射为稠密向量，加上位置编码。但 Decoder-only 不需要 [CLS] 这种特殊的聚合 Token——它通常直接在序列末尾做预测。

**第二步：特征解码。** 多层 Decoder Block 堆叠。每一层内部——因果自注意力让每个 Token 只能看到前面（包括自己）的 Token，不能看到后面的 Token。这是通过一个「下三角掩码矩阵」实现的——在对注意力分数做 Softmax 之前，将未来位置的分数设为 -∞（Softmax 后这些位置的权重变为 0）。经过多层处理后，每个 Token 的向量融合了它之前的所有上下文信息。

**第三步：输出生成。** 取最后一个 Token 在最后一层的输出向量，送入一个线性层（将隐藏维度映射为词表大小 |V|）+ Softmax，得到下一个 Token 的概率分布。根据这个分布采样出一个 Token。采样出的 Token 拼回输入序列的末尾——形成新的输入——再重复上述过程——这称为「自回归生成」。

Decoder-only 的极简设计带来了几个关键优势：没有 Encoder → 省掉了约一半的参数量和计算量 → 在同样的算力预算下可以堆叠更大的 Decoder → 更强的表达力；架构更统一 → 训练和推理的工程实现更简单 → 分布式扩展更容易。这正是 Decoder-only 最终成为大语言模型主流架构的根本原因。


## 第 8 页 · Encoder-Decoder 架构详解

**PPT 内容**：
- Encoder-Decoder 架构同时选取了 Transformer 中的编码器和解码器部分，并采用交叉注意力机制来实现两者之间的交互。
- 完整的架构包含五个部分：编码输入嵌入 → 编码特征编码 → 编码后固定长度的上下文向量 → 解码输入嵌入 → 解码模块中用交叉注意力引用编码器输出。

### 逐页讲解

Encoder-Decoder 架构是 Transformer 原始论文的完整形式。它有五个组成部分，比 Encoder-only 和 Decoder-only 都更复杂：

**部分一（编码输入嵌入）**：源序列的 Token 经嵌入矩阵映射为向量，加位置编码。例如在翻译任务中，这是「英文句子」的词向量。

**部分二（编码特征编码）**：多层 Encoder Block 堆叠——和 Encoder-only 架构完全一致——每个 Block 包含完全双向自注意力 + 前馈网络。每个位置的输出向量深度融合了整个源序列的全部上下文（上下左右所有 Token 的信息）。

**部分三（上下文向量）**：编码完成后，最后一层 Encoder 的输出是一个维度为「源序列长度 × 隐藏维度」的向量矩阵。这就是整个源序列的「完整语义编码」。Decoder 下一步就可以把它当作固定参考信息持续查询。

**部分四（解码输入嵌入）**：目标序列的已生成部分经嵌入矩阵映射为向量，加位置编码。例如翻译任务中，这是「已经翻译出来的中文字」的词向量。

**部分五（解码 + 交叉注意力）**：这是 Encoder-Decoder 架构最精妙的部分。Decoder 的每一层包含三个子层：①因果自注意力（确保生成时不被未来 Token 干扰）；②**交叉注意力（Cross-Attention）**——这里的 Query 来自 Decoder 当前层的自注意力输出（「我当前正在生成什么内容」），Key 和 Value 来自 Encoder 的最终输出（「源文本的完整含义」）；③前馈网络。交叉注意力的作用是——在生成每一个目标 Token 时，Decoder 都能「回头看」源序列的全部语义，从中抽取和当前生成相关的信息。这比单纯的 Prompt 前缀拼接（Decoder-only 的做法）更深入。

Encoder-Decoder 的核心优势在于它分别在编码端和解码端优化了理解能力和生成能力。在机器翻译、文本摘要等「输入 → 输出」模式明确的任务上，它通常能用更少的参数达到比 Decoder-only 更好的效果。但代价是架构更复杂、需要维护两套参数（编码器+解码器），在极大规模（百亿以上参数）时这一开销就越来越不经济。


## 第 9 页 · 三种架构的注意力矩阵差异

**PPT 内容**：
- 三种架构虽然在宏观上都源于 Transformer 框架，但在注意力矩阵上的差异是本质性的。
- Encoder-only：「完全」的注意力——每个 Token 能看到序列中的所有其他 Token（双向）。
- Decoder-only：「下三角」的注意力——每个 Token 只能看到自己以及之前的 Token（单向/因果）。
- Encoder-Decoder：「完全 + 下三角」的结合——编码器用完全注意力双向理解输入；解码器用下三角注意力因果生成输出；解码器还通过交叉注意力引用编码器的全部输出。

### 逐页讲解

⭐ 这一页揭示了三种架构差异的**数学本质**——注意力掩码的形状。

想象一个 N 个 Token 的序列。自注意力机制会计算一个 N×N 的注意力分数矩阵，其中第 (i, j) 个元素表示「Token i 应该给 Token j 多少关注」。掩码矩阵决定了哪些位置之间的关注是被允许的：

**完全注意力（Encoder-only）**：掩码矩阵全是 1——任意 Token i 和 Token j 之间都可以互相关注。从 i=1 到 i=N 的所有位置都能看到所有其他位置——包括前面的和后面的。这使得编码器能捕捉到最丰富、最完整的双向上下文信息。例如理解句子「他把他最喜欢的书送给了一个朋友」时，「他」和「朋友」的关系可以双向互动——「他」告诉「朋友」我是送东西的人，「朋友」告诉「他」你是被赠送的对象。

**下三角注意力（Decoder-only）**：掩码矩阵是下三角矩阵——对角线及以下为 1，对角线上方为 0（被设为 -∞）。位置 i 只能关注位置 j ≤ i。这是为了保证自回归生成的**因果一致性**——在预测第 5 个 Token 时，模型不能「偷看」第 6 个 Token 的内容（因为在实际推理时第 6 个 Token 还没生成）。这种约束既是限制（牺牲了双向理解），也是必要（确保生成过程的逻辑一致性）。

**完全 + 下三角（Encoder-Decoder）**：编码器内的自注意力是全完掩码（双向）；解码器内的自注意力是下三角掩码（因果）；解码器的交叉注意力则完全不受因果限制——解码器的每个生成位置都可以用到编码器的整个输出（完全掩码）。这实现了「理解上的双向、生成上的因果」的精确分工。

这三种掩码形状直接决定了模型的输入处理方式和输出任务天然倾向性——也是选择模型架构的最重要的元决策依据。


## 第 10 页 · Encoder-only 架构在任务上的倾向

**PPT 内容**：
- 由于独特的模型设计和注意力矩阵差异，在同等参数规模下不同架构各有任务倾向。
- Encoder-only 架构中的双向注意力允许模型充分考虑序列中的前后文信息，捕捉丰富的语义和依赖关系——因此在情感识别等**判别式（理解型）任务**上表现优异。
- 然而——缺少解码器组件，无法直接输出序列。所以不适用于对话问答等**生成式任务**。

### 逐页讲解

这一页详细分析了 Encoder-only 架构「擅长什么」「不擅长什么」。

**擅长——理解/判别型任务**。Encoder 的双向注意力是理解任务的最优配置。情感分析（判断一段文本的情感倾向——正面/负面）；文本分类（主题归类、垃圾邮件检测）；句子对推理（判断两句是否互为同义改写、是否构成蕴含关系）；命名实体识别（找出一句话中的人名、地名、组织名）。所有这些任务的本质都是——给定完整的文本，提取出结构化的理解结果（一个标签、一个类别、一组实体边界）。Encoder 的双向注意力让模型能在完整的上下文中判断每个部分的确切语义——这是单向模型做不到的。

**不擅长——生成型任务。** 因为 Encoder-only 没有 Decoder，它不能逐个 Token 地生成输出文本。它只能输出一个标签、一个类别、或者对每个位置打一个分类分数。你无法让它「写一篇论文」或「回答一个开放问题」——它根本不知道如何一字一字地生成新内容。这在生成式 AI 时代的竞争中是一个致命短板——对话、写作、代码生成、翻译这些最受关注的应用全都需要生成能力。

这也是为什么 BERT（Encoder-only，2018）在提出时横扫了几乎所有的 NLP 理解 benchmark，但在 2020 年后 GPT 系列（Decoder-only）逐渐走上舞台中央——因为生成能力更面向真实世界。


## 第 11 页 · Encoder-Decoder 架构在任务上的倾向

**PPT 内容**：
- Encoder-Decoder 架构通过添加解码器来基于编码器输出的上下文表示逐步生成输出序列——因此既能做判别也能做生成。
- 但解码器的加入也导致训练和推理成本的增加。

### 逐页讲解

Encoder-Decoder 架构是「通用型选手」——兼具理解能力和生成能力。Encoder 负责深度理解输入，Decoder 负责生成输出，交叉注意力将两者的信息精确地连在一起。这种设计在需要「强输入-输出映射」的任务上表现尤其突出：机器翻译（Encoder 深入理解源语言句子→Decoder 生成目标语言句子→交叉注意力确保翻译每一个词时都能精确引用到源句对应部分）；文本摘要（Encoder 读完长文→Decoder 输出精炼摘要）；语音识别（Encoder 编码音频隐状态→Decoder 逐字转换为文本）。

但这种「双全」是有代价的——**计算成本**。在同等参数量下，Encoder-Decoder 架构需要维护编码器和解码器两套参数。标准 Transformer 配置中，编码器参数约占总参数的一半。这意味 110 亿参数的 T5 模型，实际上大约只有 55 亿参数在解码器侧用于生成——而同等规模的 GPT-3（Decoder-only）的全部参数都在生成侧。因此 Decoder-only 在生成质量和灵活性上具有天然的结构优势。

这决定了架构选择的工程哲学——如果任务是「输入和输出完全对齐有明确对应关系」（翻译、摘要），Encoder-Decoder 仍是首选；如果是「通用开放域生成」「对话」「代码生成」或追求「用更大更多参数撑死一切」——Decoder-only 更经济且可扩展。


## 第 12 页 · Decoder-only 架构在任务上的倾向

**PPT 内容**：
- 大规模预训练数据的加持使得 Decoder-only 架构的模型能够生成高质量、连贯的文本。
- 但是缺乏编码器提供的双向上下文信息，这一架构在理解复杂输入数据时存在一定局限性。

### 逐页讲解

Decoder-only 架构的核心张力——**生成能力极强但输入理解相对单向**。在超大规模预训练的加持下，Decoder-only 模型即使不通过双向编码器也能通过累积上下文在单方向推断出高度合理、语义一致的输出。但是——它缺少 Encoder 提供的双向上下文——在阅读复杂输入时更依赖短距离线索而非全局结构（尤其在早期层次），且注意力为因果掩码——左侧的 Token 看不到右侧的 Token——当输入内容本身就需要跨距解读时能力受限。

尽管如此，在参数和数据量足够大后（GPT 系列，参数量从 1.17 亿扩展到 1750 亿），Decoder-only 的单向注意力劣势被大幅稀释——量变引起的质变掩盖了结构设计的细节差异。这也正是 Decoder-only 在当下压倒另外两种架构的规模证成。


## 第 13 页 · 架构的演变趋势——Decoder-only 逐渐成为主流

**PPT 内容**：
- 大语言模型的架构经历了从 Encoder-only 到 Encoder-Decoder，再到 Decoder-only 的演变，Decoder-only 架构逐渐成为主流。
- 三条走势图：Encoder-only——早期爆发、渐趋平缓 / Encoder-Decoder——蓄势待发、后发制人 / Decoder-only——稳步崛起、持久发展。
- 初始数据来源：《Harnessing the Power of LLMs in Practice: A Survey on ChatGPT and Beyond》

### 逐页讲解

这是一个极富信息量的历史纵览图——通过三条曲线的走势，直观展示了三种架构在学术界影响力此消彼长的过程。

**Encoder-only（早期爆发 → 渐趋平缓）**：2018 年 BERT 发布时横扫了几乎所有 NLP benchmark——GLUE 排行榜上 BERT 将此前最好成绩一举提升近 10 个百分点——一时之间 Encoder-only 是绝对主流。然而到 2019-2020 年，随着 GPT-2、GPT-3 展示出强大的生成能力，Encoder 的声量开始下滑。至 2022 年 ChatGPT 发布后，Encoder 路线边缘化为「非主流」。

**Encoder-Decoder（蓄势待发→后发制人→趋于稳定）**：T5（2019）曾试图打造适应生成和理解的大一统模型，BART 及多语言版本在翻译和摘要等固定输入输出任务上保持一席之地。但由于参数约束和生成灵活性被 Decoder-only 的双重挤压，逐步变为固定任务型的「特定最优方案」。

**Decoder-only（稳步崛起→持久发展）**：2020 年 GPT-3 将 Decoder-only 拉升到 1750 亿参数并公布了大量的 Few-shot 能力→业界轰动。2022 年 ChatGPT/GPT-4 的推出标志着 Decoder-only 彻底统治了公众关注和资本方向。这条曲线至今（2026 年）仍在持续爬升。

这个演变的历史逻辑是——架构「谁能更好地利用规模化」在长周期中胜出。更简洁、更集成的架构（Decoder-only）消除了复杂组件间的耦合和参数浪费，在资源总量相同的情况下能将所有计算力集中到「生成」本身——结果自然远超两个模块分开训练的架构。


## 第 14 页 · 三种架构的格局——未来未定

**PPT 内容**：
- 尽管 Decoder-only 目前已经成为主流架构，但 Encoder-only 和 Encoder-Decoder 架构仍具有重要应用价值。而且，哪种架构最终问鼎也尚无定论。

### 逐页讲解

这一页提醒我们不要「刻板化历史」。在快速演进的大语言模型领域，Decode-only 当前的领先并不代表永久正确。Encoder-only 在**双向深度理解**（如法律文本细粒度语义分析、医学文献精确信息抽取、复杂问题分解）领域仍有天然优势。Encoder-Decoder 在明确的「输入到输出映射」（翻译长文档、严格格式的摘要生成、多跳推理 QA 系统）上仍最优。

未来可能会出现的变数——人们发现超级规模的 Decoder-only 模型在「理解真实世界因果逻辑」方面始终不足，从而重新引入双向编码结构；混合架构可能有新的解决方案（类似 ViT 和 CNN 在视觉领域的协作）；硬件架构的特殊化可能更倾向某种注意力配置。

总之——当前的 Decoder-only 主导只是历史的一个阶段，不应视为绝对终点。这种开放性也鼓励学生持续关注最新进展——不要过早固化认知。


## 第 15-16 页 · Encoder-only 架构的模型家族概览

**PPT 内容**：
- 当前最常用的 Encoder-only 架构模型是 BERT 及其变体——RoBERTa、ALBERT 等。

### 逐页讲解

过渡到 Encoder-only 架构的详细介绍。在整个 156 页的 PPT 中，Encoder-only 占据了约 15 页篇幅——虽低于 Decoder-only 的约 30 页但仍是一项重要组成部分。下面将逐页解析 BERT、RoBERTa 和 ALBERT 三大系列的各项细节。


## 第 17 页 · BERT 模型架构与两个版本

**PPT 内容**：
- BERT 模型的结构与 Transformer 中的编码器几乎一致，只是规模不同。
- BERT 有两个版本——BERT-Base 和 BERT-Large。
- 参数表：BERT-Base——12 个编码模块，隐藏层维度 768，自注意力头数量 12，总参数量 1.1 亿；BERT-Large——24 个编码模块，隐藏层维度 1024，自注意力头数量 16，总参数量 3.4 亿。

### 逐页讲解

BERT 的结构精确复制了 Transformer Encoder，没有做任何架构上的改动——其创新在于预训练目标和训练策略。两个版本的差异点提供了一个典型的「小/大模型对比」：

**层数**：12 层 vs 24 层——深两倍。层数越多意味着语义信息可以经过更多层次的逐步抽象和融合——每一层都可能捕捉到不同程度和类型的语义特征（低层→词法、中高层→句法、高层→全局语义关系）。

**隐藏维度**：768 vs 1024——更宽。隐藏维度决定了每层的「信息容量」——更大的向量能编码更多的语义特征维度。

**注意力头数量**：12 vs 16——更多的并行视角。多头注意力机制让我们能在不同的隐式子空间中捕捉到不同类型的词间关系——例如有的头擅长捕捉语法关系，有的头擅长捕捉句法，有的擅长捕捉主题连贯性。头更多→视角更丰富。

**总参数量**：1.1 亿 vs 3.4 亿——大约 3 倍。这个参数规模在当下的 GPT/LLaMA 时代看非常小（LLaMA-3 最大参数 4050 亿），但在 2018 年已经是当时的最前沿规模——对比同时期较常见的 NMT 翻译系统参数量通常≤2亿已是巨大跃进。


## 第 18 页 · BERT 预训练数据集

**PPT 内容**：
- BERT 使用小说数据集 BookCorpus（包含约 8 亿个 Token）和英语维基百科数据（包含约 25 亿个 Token）进行预训练，总计约 33 亿个 Token，15GB 左右的数据量。

### 逐页讲解

BERT 的训练数据量—33 亿 Token（~15GB）——在 2018 年非常庞大。BookCorpus 和英语维基百科代表了两种互补的数据类型——虚构叙事文（长篇章、丰富的叙事结构、细腻的人称与情境）和服务于事实知识的百科文（结构性的、主题化的、浓缩的）。混合给 BERT 提供了涵盖叙事性上下文和事实知识信息的双向视野。但总量只有 33 亿 Token，是今天的大模型（如 LLaMA-3.1 的 15 万亿 Token）的几千分之一。这解释了为什么以今天的标准看——BERT 在很多长尾知识点上近乎空白。


## 第 19 页 · BERT 预训练任务——MLM 和 NSP

**PPT 内容**：
- BERT 开创性地提出掩码语言建模（Masked Language Model, MLM）和下文预测（Next Sentence Prediction, NSP）两种任务来学习生成上下文嵌入。
- MLM——掩码语言建模：根据前后文完成完形填空（「今天天气 [MASK]，适合去 [MASK]」→ 晴、公园）。
- NSP——下文预测：判断两个句子是否连续（给定两句 A 和 B，判断 B 是否是 A 的下一句）。

### 逐页讲解

BERT 的预训练包含两项任务，它们合在一起能带来两个维度的训练信号——词语层面 + 句子层面的语义关系均被建模。

MLM 是 BERT 的最大贡献——不像之前的语言模型（GPT）只能从左往右单方向预测下一个词，BERT 随即将整个句子中的 15% token 随机遮挡，然后训练模型借助上下文（左边+右边都有信息）预测被遮的词。这迫使其理解了词的语境本质——同一个词（「苹果」），在食物语境（"他吃了 [MASK]"）和科技语境（"[MASK] 发布了新手机"）——模型要根据周边未被遮的词推断——从而使 BERT 学会了对词语进行**语境化消歧**。这也是「完形填空」——而不再是「预测下一词」。

NSP 辅助训练句子级语义关系——从文档里随机抽两句——如果这两句在原文中刚好相邻→正例（标签=1）；如果是从不同文档随机抽取→反例（标签=0）。这帮助了 BERT 在后续句对任务（如文本蕴含/自然语言推理——"A 蕴含 B 吗？"）上拥有更强的句子级基线。但后来 RoBERTa 通过消融实验发现 NSP 其实作用不大——甚至可能干扰核心的 MLM 训练效果。


## 第 20 页 · RoBERTa 简介——大力出奇迹

**PPT 内容**：
- 为了解决 BERT 训练不充分的问题，RoBERTa 在 BERT 的基础上使用更大的数据集、更长的训练时间、更细致的学习任务和超参数调整来优化预训练过程。
- 对比：包含 160G 的训练数据（vs BERT 15G）/ 动态 Mask（vs BERT 静态 Mask）/ 移除 NSP 任务（vs BERT 添加 NSP）/ 更长的训练步数和更大的 batch size。

### 逐页讲解

RoBERTa 的全称是「Robustly optimized BERT approach」——翻译过来就是「把 BERT 的训练做得更坚固、更充分、更到位」。RoBERTa 没有修改任何 BERT 的模型架构（层数、维度、注意力头数和 BERT 完全一致），但是做了几个关键性的训练优化：

**更大量数据**：从 15GB 剧增到 160GB（接近 11 倍），新增了新闻新闻 CC-News（~76GB），网页开放数据集 OpenWebText（~38GB），故事数据集 Stories（~31GB）等多个大规模、高多样性的来源。

**更长训练步数**：从原 BERT 的 100K 步扩展到 500K 步，确保更充分的学习。

**动态掩码**（下两页详述）：每次数据加载时重新随机生成掩码，而不是离线预先做好静态掩码再反复训练。

**移除 NSP 辅助任务**：消融实验表明 NSP 对下游任务帮助很小甚至会干扰核心 MLM，所以 RoBERTa 果断把它砍了。

RoBERTa 的核心理念被业界总结为——「**训练不充分是大模型性能的最大隐性瓶颈**」——这一结论为后来的 Scaling Law 提供了实证基础。大力出奇迹在实际中极其有效。


## 第 21 页 · RoBERTa 模型结构

**PPT 内容**：
- RoBERTa 在模型结构上与 BERT 基本一致，都是由多个编码模块堆叠成的。同样有两个版本：RoBERTa-Base（12 编码模块，768 隐藏维，12 自注意力头，1.2 亿参数）基本对标 BERT-Base；RoBERTa-Large（24 编码模块，1024 隐藏维，16 自注意力头，3.5 亿参数）基本对标 BERT-Large。

### 逐页讲解

这一页对比了 RoBERTa 和 BERT 在结构上的**完全相同**——12 层/24 层，768/1024 维，参数量 1.2/3.5 亿仅是微小的差异（可能是类嵌入或其他非常细致的不同参数计入方式）。这是 RoBERTa 最想传达的信息——「我不需要特殊的架构魔法，我只需要把训练做足」。这个教训深刻地影响了大语言模型时代的所有研究——足够大的数据和足够久的训练常常比精巧的结构创新带来的涨幅更大。


## 第 22 页 · RoBERTa 预训练数据集

**PPT 内容**：
- RoBERTa 在 BERT 的基础上添加了新闻数据集 CC-News（76GB）、网页开放数据集 OpenWebText（38GB）和故事数据集 Stories（31GB），总数据量约 160GB。

### 逐页讲解

数据集扩展的三个方向——新闻提供了时效性和事实权威性（对 NLP 下游事实性问答和信息抽取有很大利好），OpenWebText 提供了大量网络用户的自然表达（能够涵盖从学术讨论到随手日志的各种文风——提升语言模型对多风格的泛化能力），故事（Stories）填充了叙事链和长篇情节中的时序敏感度和人物关联等叙事理解能力。这个扩展策略证明了——**数据的多样性比单纯的更多同类数据更有价值**——后来所有大模型在选择数据混合时都沿用了这个准则。


## 第 23 页 · RoBERTa 预训练任务——动态掩码

**PPT 内容**：
- RoBERTa 移除了 BERT 中的下文预测任务（NSP），并将掩码语言建模任务强化为**动态掩码**语言建模——增加模型训练的多样性，帮助模型学习到更丰富的上下文信息。
- 静态掩码：在预处理阶段就离线把数据分成 10 个副本分别做好固定的掩码（每个副本内各样本框一样的 Mask 位置不变）——训练时就着这些固定掩码进行。
- 动态掩码：每次将数据送入大模型前再当场随机生成掩码——同一组句子在每一轮训练看到时被遮住的词都不相同（掩码位置随机变化）→增加训练多样性。

### 逐页讲解

⭐ 这一对比非常重要——**静态 vs 动态掩码**。原 BERT 的预处理流程——先把训练数据拷贝 10 个副本，每个副本离线统一随机遮好 Mask 位置→训练时始终读固定的这 10 种遮法。这本是为了训练加速而牺牲多样性的工程妥协——同一句子在若干 epoch 中反复看到的都是同样的那几种遮法。

RoBERTa 放弃了这 10 个静态副本——每次加载数据到 batch 时动态生成该 batch 的随机掩码——这意味着同样的一句话在不同的 epoch 中被遮的单词完全不一样——模型被迫从各种不同的上下文中综合推断被遮的词——极大地增加了训练数据的有效利用率和消歧能力。这相当于 **「免费的数据增强」**——不需要增加实际样本，但通过创建不可预知的语境遮蔽变化，迫使模型学到更鲁棒、更灵活的语义表示。


## 第 24 页 · RoBERTa 总结

**PPT 内容**：
- RoBERTa 在文本分类、情感分析、问答系统等很多自然语言处理任务上都取得了非常好的效果。相比 BERT 具有更高的性能和更好的泛化能力。但也存在问题——模型训练时间更长，训练成本更大。

### 逐页讲解

RoBERTa 的成功证明了「大力出奇迹」这条路的有效性——架构不变仅改进训练策略就可以把性能提一大截。这个结论深刻影响了后续所有的预训练研究——让研究者开始把注意力从「发明新的神经网络结构」转向「优化训练策略、扩大数据规模、延长训练时间」。但这也意味着训练成本显著提高——需要更大的算力、更长的时间——这直接引出了后续的 Scaling Law 研究（Optimal vs 最佳性价比之间的对峙）。


## 第 25 页 · ALBERT 模型结构——轻量化 Encoder

**PPT 内容**：
- 为了提高训练和推理效率，Google 团队提出了更轻量化的 ALBERT 模型。其结构基本不变，但其独有的参数因子分解和跨层参数共享显著降低了模型参数量。配图：BERT（大）vs ALBERT（小）——架构不变，规模显著降低。

### 逐页讲解

ALBERT 的立意与 RoBERTa 完全不同——RoBERTa 要**更强**，ALBERT 要**更轻**。参数量大=占用显存大=训练和推理花费高=难于在有限资源中部署。ALBERT 的轻量化策略把总参数量压缩到了极低值——但同时保持了和 BERT 近似的性能水平（计算量并不因此减少）。这个方向对于当下的大模型微调和成本控制有非常重要的启示（后续的 PEFT 本思想的进化）。接下来两页详述两种压缩技术。


## 第 26 页 · 模型参数来源——Embedding 和 Encoder 各占多少

**PPT 内容**：
- 对于 Encoder-only 架构的模型而言，参数主要来源于词嵌入矩阵以及 Encoder 块。前者约占整体参数的 20%，后者约占 80%。
- 词嵌入矩阵+Encoder 块是参数集中区——这两项是后续优化压缩的主要重攻击点。

### 逐页讲解

ALBERT 设计的洞察点——BERT 的总参数量大约 **20% 在词嵌入矩阵**、**80% 在 Transformer Encoder blocks**。词嵌入矩阵参数量 = 词表大小 V × 隐藏维数 H。以 BERT 为例——V≈3 万，H=768 → 词嵌入矩阵参数 ≈ 2300 万。

Encoder 块的参数量则主要来自每个块内的多头注意力投影矩阵（W_Q, W_K, W_V, W_O）以及 FFN 的前馈矩阵。这部分参数与 H² 成正比——隐藏维每增一倍，块内参数量膨胀为原来的四倍。

了解这个参数分布比例，就知道参数压缩应该同时针对两个源——一是 Embedding 压缩（因子分解），二是 Transformer block 重用（跨层参数共享）。


## 第 27 页 · 参数因子分解——Embedding 矩阵的瘦身法

**PPT 内容**：
- ALBERT 采用参数因子分解的方式减少词嵌入模块的参数量。
- 词嵌入矩阵 E ∈ R^{V×H}——参数量 = V×H = 30000×768 = 23,040,000。
- 分解后——E1 ∈ R^{V×E} + E2 ∈ R^{E×H}。其中 E=100（中间维度，远小于 H=768）。
- 总参数量 = V×E + E×H = 30000×100 + 100×768 = 3,076,800——从 2300 万降至 307 万，缩减约 87%。

### 逐页讲解

⭐ 参数因子分解的数学原理非常简单——把一个大矩阵拆成两个小矩阵的乘积。

原词嵌入矩阵是 V×H。现在拆成两步——先把 Token 索引映射到一个低维中间表示（V×E），再从低维中间表示升到最终的隐藏层维度（E×H）。对 BERT——原约 2300 万→降至约 300 来万。这个中间维度 E（如 100）起到了信息瓶颈的作用——同时也是大幅压缩参数量级的关键。

这种拆分技术的灵感来自矩阵分解中的低秩近似——不显式构造全秩、不存储高维的完整 V×H 矩阵——用一个低秩瓶颈维度串联两步实现了近似效果。ALBERT 在词嵌入侧节省的参数量占总模型的大约 18%——是一个简单但高效的解法。


## 第 28 页 · 跨层参数共享——Encoder 块复用

**PPT 内容**：
- 对于 Attention 与 FFN 模块，ALBERT 采用跨层参数共享的方式减少其参数量。所有 Transformer Block 共享同一组参数——即「一个 block 的参数，被全部层复用 12 次」（或 24 次）。配图——多层 block 的参数箭头循环指引至同一组参数。

### 逐页讲解

这是 ALBERT 最激进的参数压缩措施——**所有 Transformer 层的参数完全相同**。原 BERT 每层都有自己独特的一套权重（12 层 = 12 套独立参数）。ALBERT 改为——只训练一层 Transformer Block，然后这层参数在所有的 12 个位置被循环复用（像 RNN 在时间步上复用一样）。

因此 Encoder block 的总参数量变成了 BERT 的 1/12（ALBERT-Base）或 1/24（ALBERT-Large）。这个措施的代价是——总计算量**并没有减少**——虽然在参数量上 12 层共享——但前向传播仍需将这组共享参数在 12 个层中各算一次——也就是说推理的 FLOPs 和 BERT 一样多。参数少——训练和存储的显存、硬盘空间大幅节约，但前向推理速度并不加快。

这种策略还有一个额外的好处——跨层参数共享天然地起到了正则化作用——防止多层累加产生的过拟合（一些下游小数据任务上反而效果更好或同等）。它表明了「额外的独立性参数并非永远有用」这一核心观点。


## 第 29 页 · ALBERT 的预训练任务——用 SOP 替换 NSP

**PPT 内容**：
- ALBERT 采用了和 BERT 完全相同的数据集，但在预训练任务上，ALBERT 保留了 BERT 的掩码语言建模任务，而将下文预测任务（NSP）替换为句序预测（Sentence Order Prediction, SOP）。
- SOP——判断两个句子是正序的还是倒序的（给定 A 和 B，判断是「A→B」还是「B→A」原顺序谁在前？）。配图例子：两段文本「真不是这样啊我们山东人」「瞎说啥呢，你这是!」——与「瞎说啥呢你这是!」在前的反序——SOP 需判断。

### 逐页讲解

句序预测（SOP）是对 NSP 的一个重要改进。NSP 判断「两句是不是连续」——这个任务负面样本来自完全不同的文档——因此模型可能仅靠明显的「主题是否一致」来区分正反例，而非真正理解两句之间的精细连贯关系。

SOP 更聚焦于**句子级的逻辑顺序**——同一个上下文中两句调换顺序——正例是真实顺序，反例把 B 放到 A 之前。这更难——要求模型理解两句之间事件的时序性、因果性——比如「A: 下雨了，B: 我带了伞」——B→A 时序不符合常识（带伞是在下雨之后）。SOP 因此对很多需要理解多句顺序的下游任务更有帮助。

实验表明——SOP 在多项逻辑推理和阅读理解的下游任务中显著优于 NSP。


## 第 30 页 · ALBERT 总结

**PPT 内容**：
- ALBERT 对比其他 BERT 系列模型参数量更小、训练速度快。但它在一些任务上的表现不如 BERT。
- 优点——轻量级、模型参数量小、训练速度快。缺点——在某些特定任务上表现不佳。

### 逐页讲解

ALBERT 在轻量化这个维度上交出了极致的答卷——参数因子分解+跨层共享的结合让模型可以放到以前无法容纳的小型设备上。训练速度也因为需要更新的参数量大幅降低而有提速。但缺点不可否认——压缩得太狠了（所有层完全一样的参数），在层数很多的、需要语义区分极细微的复杂任务中，表现确实不如每层有独立参数的原始 BERT（这也是层独立和跨层共享之间的本质张力）。后续研究者更倾向于**不共享全部层但用中间粒度的共享**来保留一部分层间差异。


## 第 31 页 · Encoder-only 架构的优势

**PPT 内容**：
- 得益于双向编码模型中的全面注意力机制，基于 Encoder-only 架构的模型在需要深度理解的任务中展现出了卓越的能力。
- 全面的注意力机制——双向编码模型融合了从左往右的正向注意力以及从右往左的反向注意力，能充分捕捉每个 Token 的上下文信息。

### 逐页讲解

双向编码的优势——每 Token 能看到它所在句子的所有其他 Token（前后可见），因而对每个单独词语的理解是整个句子上下文协同消歧的结果——而非只能看到左侧信息的单向消歧模型。在需要精确理解语义细节的任务——法律条款分析、医学文本实体关系提取、细粒度情感分析——Encoder-only 的完全双向注意力让信息没有任何方向的缺失。然而这个优势在生成能力上完全不存在——Encoder 框架无法自回归地生产新文本——因此它在 AI 工厂中仅限于充当理解专用模块。


## 第 32 页 · Encoder-only 架构的问题

**PPT 内容**：
- 但由于没有寻求参数量的突破，且只专注于判别任务，难以应对生成式任务。因此在生成式人工智能中可以发挥的作用相对有限。
- 两大问题——模型体量小（最大仅到约 3.5 亿参数 vs GPT-3 1750 亿） / 专注于判别任务（无法生成开放文本）

### 逐页讲解

Encoder-only 在当下的衰落——主要靠两条腿——第一是参数量封顶太低（BERT-Large 约 3.4 亿—— RoBERTa ≈3.5 亿—— ALBERT 更小）——而对比大语言模型参数量范围数十亿到千亿甚至万亿——这个体量完全无法在通用能力上与后者竞争。第二是只做判别——情感、分类、问答等判别型任务——与生成式 AI 的未来画像缺席，而用户需要的是能说话、能写代码、能编程、能翻译——全部都需要生成能力。因此在当前抢占话语权和资本关注的生成式 AI 浪潮中——Encoder-only 仅扮演配角或作为特定领域理解模型的角色。


## 第 33 页 · 板块标题过渡——Encoder-Decoder 架构

**PPT 内容**：
- 板块标题页——基于 Encoder-Decoder 架构的大语言模型

### 逐页讲解

过渡页——从第一小节（Encoder-only）正式进入第二小节（Encoder-Decoder 架构）。


## 第 34 页 · Encoder-Decoder 架构的动机——弥补 Encoder 生成不足

**PPT 内容**：
- Encoder-Decoder 在原先 Encoder-only 架构的基础上添加一个 Decoder 组件，使其能生成连贯的输出。从而缓解了 Encoder-only 架构在生成式任务上的局限性。
- 编码器——输入文本→理解输入→生成输出→输出文本→解码器。

### 逐页讲解

为了解决 Encoder-only 的生成缺失——Encoder-Decoder 在编码器之上添加了解码器——而解码器是专门负责「逐词输出」的自回归组件。这样整个框架变为——编码器把输入压缩为语义层面的抽象表示；解码器基于这个表示在生成空间中一步步拼出目标输出序列。这条管线让模型既能消化复杂输入，又能构建任意长的输出文本序列——是弥补纯 Encoder 缺陷的最直接也是最有效的方案。


## 第 35 页 · Encoder-Decoder 架构两大部分

**PPT 内容**：
- Encoder-Decoder 架构主要包含编码器和解码器两部分。
- 编码器：由多个编码模块堆叠而成，用于将输入序列转变为固定长度的上下文向量。包含自注意力模块和全连接前馈模块。
- 解码器：由多个解码模块堆叠而成，基于上下文向量自回归地生成输出序列。包含自注意力模块、交叉注意力模块和全连接前馈模块。

### 逐页讲解

对这两大组件的模块对比——编码器模块中含双边全注意力自注意层（允许每个 token 看全部输入 token——双向充分理解）+ 前馈网络；解码器模块在编码器基础上多了个交叉注意力子层——使其在自推生成每个新 token 时，可以查阅编码器预先提取的所有输入信息。这个交叉注意力子层是所有 Encoder-Decoder 模型的生成组件与 Decoder-only 模型的最大模块区别。


## 第 36 页 · 注意力模块——三种并存的注意力

**PPT 内容**：
- 完整的 Encoder-Decoder 架构中共包含三种注意力模块：
- 自注意力模块（编码器）——编码器内部的双向完全注意力——负责理解输入序列
- 自注意力模块（解码器）——解码器内部的因果下三角注意力——负责保证生成次序和连贯
- 交叉注意力模块——解码器查询编码器输出的注意力——负责确保生成内容与输入对齐

### 逐页讲解

三种注意力在完整的 Encoder-Decoder 架构中各司其职，形成了非常清晰的分工。每个解码器层顺序执行——首先因果自注意力（保证当前 token 的输出不能看到未来的词）→然后交叉注意力（以编码器输出作为外部知识库进行查阅 → Query 来自解码自注意力输出，Key/Value=编码器输出）→最后前馈变换。交叉注意力是这个系统的精髓——编码器把完整输入（无论长短）精炼为全局上下文记忆，解码器在逐步往外说话的每一刻都可以回到这个全局记忆中引用对应当前最相关的输入信息片段。


## 第 37 页 · 三种注意力差异细节

**PPT 内容**：
- 编码器自注意力——完全注意力（所有 token 互看）
- 解码器自注意力——下三角因果注意力（只看前面，不偷看未来）
- 交叉注意力——解码器对编码器输出的完全注意力（K, V = 编码器输出，没有因果限制）

### 逐页讲解

三种注意力的视觉化比较：编码器注意力 = 全连接矩阵（所有 token 相互之间无限制交互）。解码器自注意力 = 下三角掩码（对角+左下，右上半块全为 0——因被遮成 -inf → softmax 后=0）。交叉注意力——K 和 V 来自编码器输出，Query 来自解码器上一步的自注意输出，行数=解码器序列长度，列数=编码器序列长度。K/V 是被查阅的「参考资料」——不需要因果限制（可以查阅任意位置的参考信息，无论生成了多少 token，可同时看源文的首尾）。


## 第 38 页 · 双注意力合作——理解与生成的协同

**PPT 内容**：
- 在自注意力机制和交叉注意力机制的共同作用下，Encoder-Decoder 架构不仅能够深入理解输入序列，还能够根据不同任务的需求灵活生成长度适宜的输出序列。
- 自注意力确保输入和生成内部各自的一致连贯性；交叉注意力确保生成与输入高度相关的正确输出。

### 逐页讲解

这两种注意力的结合实现了「全局理解 + 精细生成」。编码器自注意力保证源序列的每部分在生成前已经彼此充分理解——没有一词会被孤立对待。解码器自注意力保证生成的文本自身保持流畅且有逻辑；交叉注意力保证生成的每个 token 在决定时都以全局源语义作为锚——从而大幅减少了「翻译到一半脱离原文原意」或「摘要走题」等生成式故障。


## 第 39 页 · BART 语言模型——去噪自编码器 Paradigm

**PPT 内容**：
- BART（Bidirectional and Auto-Regressive Transformers）是由 Meta AI 研究院在 2019 年提出的 Encoder-Decoder 架构模型，旨在通过多样化的预训练任务来提升模型在文本生成和文本理解任务上的表现。
- 包含五项破坏文本重建任务——Token 遮挡任务、Token 删除任务、句子打乱任务、文档旋转任务等。

### 逐页讲解

BART 的本质是**去噪自编码器**——在 Encoder 输入被完整破坏的文本（被随机遮词/删词/打乱顺序/全文翻转），训练整个模型通过 Encoder→Decoder 能恢复出原本干净的文本序列。这使得模型对语法的结构极为敏感——编码器会努力在被完全打碎的重重迷雾中恢复出原文的逻辑框架；解码器则学习把这种框架转回连续的文本输出。这一训练范式天然适合所有需要从「输入」（可能不完整/噪声/压缩/低质）生成「输出」的结构化文本的任务——例如文本摘要、翻译、改写、纠错等。


## 第 40 页 · BART 模型结构

**PPT 内容**：
- BART 的模型结构与原始的 Transformer 模型相同——编码器和解码器都由多个模块堆叠而成。两个版本——BART-Small（编/解模块 6 个，隐藏维 768，自注意力头 12，总参数量 1.4 亿）；BART-Base（编/解模块 12，隐藏维 1024，自注意力头 16，总参数量 4 亿）。

### 逐页讲解

BART 结构遵循标准 Transformer Encoder-Decoder。6/12 层编码器和解码器各自独立，中间通过交叉注意力链接。总参数量比 BERT 稍大——因为 BART 包含了两套模块（编码+解码）。BART-Small (1.4亿) 对标 BERT-Base (1.1亿) 的资源量级——BART-Base (4亿) 则超越 BERT-Large (3.4亿)。这反映出编码器-解码器模型维持同等深度的编-解码双份开销。


## 第 41 页 · BART 预训练任务——五项破坏重建

**PPT 内容**：
- BART 使用了与 RoBERTa 相同的语料库来作为预训练数据，并提出了五项破坏文本重建任务来锻炼模型对文本结构和语义的深入理解。

### 逐页讲解

五项破坏重建任务的完整体系——（1）Token 遮挡——BERT 同类的随机替换为[MASK]——传统的完形填空；（2）Token 删除——每个 token 以一定概率直接删除不再出现在被破坏序列中→模型必须猜测被删词在序列中存在于何处并还原完整文段；（3）句子打乱——输入文档随机打乱句子顺序→模型按正确逻辑重新排出顺序→理解篇章结构；（4）文档旋转——选取一个随机 token 作起点，序列全文旋转一段（前半被移至末尾或其他操作）→恢复原自然顺序→训练长距离句法连贯敏感度。这组综合任务让 BART 极具泛化到多种文本修复和生成任务的能力——比 BERT（仅两种任务）和 T5（仅 span 破坏）更多元。


## 第 42 页 · BART 的应用

**PPT 内容**：
- 通过进一步对 BART 进行微调，可将其在预训练阶段学到的语言知识迁移到具体应用场景中。针对不同任务设计不同的微调方式，BART 能够适应各种 NLP 任务。
- 例如：文本翻译模型——将 BART Encoder 输入源语言，Decoder 生成目标语言；对话系统模型——在对话历史上微调 BART 使 Decoder 生成下轮响应；情感分析模型——在 BART 编码器输出上接分类头。

### 逐页讲解

BART 的适配灵活性——因为 Encoder-Decoder 架构既能理解输入又能生成输出——同一个模型微调后可以变身翻译机、对话机器人、情感标签提供者等。微调时所有参数都被更新（全量微调）——从而让模型在不同任务上获得优秀但仍然是全参数计算消耗型的特性。BART 的这种「一型多用」灵活性对于开发者来说非常有吸引力，但也正是其全量微调的不经济和高内存占用在后续催生了参数高效微调技术。


## 第 43 页 · T5 语言模型——Text-to-Text 大一统范式

**PPT 内容**：
- 同年，谷歌探索了 T5（Text-to-Text Transfer Transformer）范式——通过构造合理的输入前缀，实现了各种 NLP 任务的大一统。
- 此前为每类任务单独设计建模和微调方案（翻译用序列到序列→情感分类用 MLP 头→总结用另一个编码器解码器），耗时且低效。T5 将所有 NLP 任务统一转换为 **Text-to-Text** 格式——输入有前缀描述任务类型——输出目标文本，一一通过同一套模型和训练目标解决。

### 逐页讲解

⭐ T5 这「Text-to-Text」思想非常精妙——统一了所有自然语言处理任务格式——「翻译成英文：今天天气怎么样？」→'How is the weather today?'；「总结以下内容：(长文)」→'短文总结……'；「情感识别：你这次作业完成地很不错。」→'积极情感'。所有输入都是统一的前缀 + 文本；所有输出统一都是目标文本字符串（并不是浮点数类别概率向量）。这打通了所有 NLP 任务在技术上的共享——全套系统仅一个 Transformer 模型即可涵盖。同时，要求输出为自由的文本串（即使是解决'分类'任务）也让其天然掌握了在语义层输出而非固定类号的人工解读。T5 的这套输入-输出设计预示了后期 Prompt 工程的逻辑基础——让人类用自然语言指令定义任务，不需定制的分类头。


## 第 44 页 · T5 模型架构与版本

**PPT 内容**：
- T5 模型的架构与标准的 Transformer 相同，但根据具体参数共发布了五个版本——T5-Small（6 模块/512 隐维/8 头 0.6 亿参数）、T5-Base（12/768/12 2.2 亿）、T5-Large（24/1024/16 7.7 亿）、T5-3B（24/1024/32 28 亿）、T5-11B（24/1024/128 110 亿）。

### 逐页讲解

T5 从大到小覆盖了全参数谱系。关键点——即使版本最大的 T5-11B（110 亿参数），其编-解两组件各自约 55 亿参数——解码能力等于编-解之间的平衡规模。Decoder-only 的模型（GPT-3）把所有参数用于生成——所以当参数博弈到千亿级别时 Decoder-only 开始彻底超越 Encoder-Decoder 在生成和文本深度获取上的平衡跷跷板。T5 一度是 Encoder-Decoder 大一统范式的最强代表。


## 第 45-48 页 · T5 训练数据及任务

**PPT 内容**：C4 数据集对 Common Crawl 清洗过滤后约 750GB。Span Corruption 预训练——每次随机遮蔽连续一段 token（span）——让模型以完形填空的方式复原整段 span 的内容——比单个 token 掩码需要更深层理解句子结构和语义。

T5 下游任务适配——不改变模型结构——通过 Prompt 前缀直接执行翻译/情感识别/语义相似度/总结等任务。

### 逐页讲解

C4 极大扩展了训练语料规模——从维基百科+小说这种精编数据到世界上互联网最庞大的网页抓取库（Common Crawl）→让 T5 掌握了大量真实互联网语言的自然变体的统计规律。Span Corruption——BERT 只遮单个 token——模型可以用周围局部词元复原（对更长语义结构的理解要求不高）；T5 的 Span 遮掉多个连续 token——迫使模型必须理解更大跨度的语义空缺和词组间的紧密协作——这对生成任务的复原能力提升尤其重要。同一模型不做重新训练，仅靠不同的 Prompt 前缀即可在不同下游任务间自由切换——是后期「一个模型解决所有问题」范式的最直接预期。


## 第 49 页 · Encoder-Decoder 架构总结

**PPT 内容**：汇总表——T5 (110亿参数/750GB语料)、mT5 (多语言版130亿/9.7TB)、T0 (零样本能力 30-110亿/~400GB)、BART (1.4-4亿参数/~20GB语料)、mBART (0.4-6.1亿/~1TB语料)。

### 逐页讲解

Encoder-Decoder 家族的总参数量普遍小于同期的 Decoder-only 大型模型——因编解码组件分离造成规模效益不如后者集中。但其在翻译、摘要等固定输入→输出映射任务上仍是行业标准。T5 的 110 亿是这个家族在预训练时期的巅峰之作——后续的 T0 进一步展示了规模与零样本能力的协同增幅仍有效。


## 第 50 页 · 板块标题过渡——Decoder-only 架构

**PPT 内容**：章节过渡——基于 Decoder-only 架构的大语言模型

### 逐页讲解

进入全 PPT 中最大的板块——Decoder-only 模型宇宙。接下来约 30 页（PPT的第 50-79 页）将深入 GPT 系列（五个发展阶段）和 LLaMA 系列（开源最强线）。


## 第 51 页 · Decoder-only 的动机——省掉编码器

**PPT 内容**：
- 开放式生成任务中输入序列通常较为简单或不明确——并不需要一个完整的编码器来消化大量结构化信息。
- 维护一个编码器所需的参数约占总模型的一半——消耗大量算力。对于输入简单的生成任务，这如同「大炮打蚊子」，性价比不高。

### 逐页讲解

Decoder-only 出现最大的工程诱因——很多生成任务（写小说、聊天、代码生成）不需要一个「庞大昂贵的专用理解器」。输入只是一小段 prompt——系统解码器本身夹带在这些 prompt 的因果链中即可单方面推演后续内容。省下编码器 → 同样计算预算下半数以上的参数都可集中在生成能力 → 同样规模下生成更强。这就是 Decoder-only 统治大语言模型的核心规模胜出逻辑。


## 第 52 页 · Decoder-only 架构的两大优势

**PPT 内容**：
- Decoder-only 架构去除了 Transformer 中的编码器部分，其简化的架构设计和强大的可扩展性，使得 Decoder-Only 架构被广泛应用于大规模语言模型。
- 模型轻量化——去除编码器后模型更加轻量，提升训练和推理速度
- 更强的可扩展性——更简单的架构更容易通过堆叠参数量来提升生成质量和能力

### 逐页讲解

Decoder-only 的两个禀赋——架构极简且集中。去掉约一半参数后模型相同算力预算下可升高层数/扩宽隐藏维度/增多注意力头——所有新增算力投入直接用在生成质量上——效率极高。架构简明使得扩展更可预测——调整层数和隐藏维时看到的效果基本呈幂律 Scaling 关系——更容易被工程界和投资界纳入产品和战略预期。


## 第 53 页 · Decoder-only 的两大代表：GPT 和 LLaMA

**PPT 内容**：
- Decoder-only 架构中最流行的是 OpenAI 提出的 GPT 系列模型以及 Meta 提出的 LLaMA 系列模型。
- GPT：起步最早、逐渐闭源、做大模型。LLaMA：起步较晚、坚持开源、提高效率。

### 逐页讲解

GPT 和 LLaMA 代表了两种截然不同的大模型哲学——GPT 路线是从 GPT-1（1.17 亿参数）到 GPT-4（未公开但推测超过万亿级别参数）——纯粹的力量叠加上限路线——一个参数就是一分能力——将模型体量做到人类经费所能及的最大。LLaMA 路线是在同等参数量下追求「更高效率」——遵循 Chinchilla 法则让模型在更小参数上获得更大数据的训练——同时在「开源」路线上构建社区生态（Alpaca/Vicuna/CodeLLaMA/LLaVA 等衍生模型大量涌现）——成了「大众可用的最强 AI」的化身。


## 第 54 页 · GPT 系列模型演进总览——五个阶段

**PPT 内容**：
- GPT 系列自 2018 年问世以来，在规模、预训练范式上不断演进，性能飞跃式地提升，引领了大语言模型的发展浪潮。
- 演进五阶段——初出茅庐 GPT-1 (2018.06, 1.17亿参数/~5GB)，小有所成 GPT-2 (2019.02, 最高15亿/40GB)，崭露头角 GPT-3 (2020.05, 最高1750亿/1TB)，一鸣惊人 ChatGPT (2022.11, 参数未公开)，大放异彩 GPT-4 (2023.03) / GPT-4o (2024.05, 参数未公开)。

### 逐页讲解

GPT 系列是整个大语言模型历史的封面故事。五阶段划分恰好对应「业界的自我认知跃迁」。GPT-1 初出茅庐——只是证明了用自回归单向语言模型做大规模预训练是可行的——能力有限但方法可行。GPT-2 小有所成——更大的 15 亿参数开始展现「无需微调直接推理某些任务」的零样本能力。GPT-3——1750 亿参数量变引发质变——涌现出了上下文学习能力——不需要额外训练只需几个示例就能理解任务并生成结果。这是一个概念性的突破。ChatGPT——将 GPT-3.5 模型商用化后以对话形态面向全体用户——引爆全行业。GPT-4——在 GPT-3基础上再次综合升级推理准确度、扩展了多模态图文输入能力。GPT-4o——全模态（视觉/语音/文本）融合+大幅降延迟。


## 第 55 页 · 初出茅庐——GPT-1 模型

**PPT 内容**：
- Transformer 的出现解决了原先困扰 OpenAI 团队很久的长距离依赖问题。与 BERT 同年，提出了第一代 GPT 模型——被称为 GPT-1。

### 逐页讲解

GPT-1 的出现几乎和 BERT 同时——2018年6月（BERT是2018年10月）。GPT-1 选择了 Decoder-only 路线——与 BERT 的双向理解设计背道而驰。GPT-1 仅约 1.17 亿参数——和 BERT-Base 的 1.1 亿相当。但其模型结构和预训练目标选择了完全区别于 BERT 的方向——它用单向自注意力（因果掩码）只允许从左到右处理输入——这与人类的书写生成习惯更贴近。尽管 GPT-1 在当时功能远不及 BERT 丰富多样（只能做补全生成——不能做情感分析、问答等理解型任务），但历史证明 OpenAI 的选择极具远见——他们从一开始就瞄准了「纯生成→通向通用人工智能」的方向。


## 第 56 页 · GPT-1 模型架构

**PPT 内容**：
- GPT-1 在结构上与 BERT-Base 高度类似，本质区别在于 BERT-Base 中的自注意力模块是**双向的自注意力**机制，而 GPT-1 中则是**带有掩码的单向自注意力**机制。配图——展示了两者注意力掩码的差异。

### 逐页讲解

GPT-1 vs BERT-Base 是同一个年代、同等规模、几乎相同的层数和维度设计的两种截然不同的哲学选择。BERT 看双向（对所有 token 能看全句任意位置）→适合「理解」；GPT 看单向（前向因果关系、不能预见后文）→适合「生成」。GPT 的掩码是一个标准的下三角矩阵——确保 Token i 只能 Attention 到 Token j ≤ i。这两个在 2018 年并列诞生的双子星——各有各的「天选使命」——最终生成的需求在整个 AI 产业发展中势头更强，因此 Decoder-only 走向统治地位。


## 第 57 页 · GPT-1 预训练方法

**PPT 内容**：
- GPT-1 使用小说数据集 BookCorpus 以及下一词预测任务来预训练，从而在不需要带标签数据的前提下学习语言「常识」。但受限于当时的训练数据量和模型参数数量，其任务泛化能力非常有限，通常需要在特定任务下进一步的有监督微调。

### 逐页讲解

GPT-1 的数据——BookCorpus（包含约 8 亿 Token 的近万本小说文本）。通过「下一词预测」（Next Word Prediction，即自回归语言模型）在无人工标注的情况下学习通用的语言统计规律。当时的局限性很明显——预训练数据在多样性和总量上远超 n-grams 之类的传统统计模型，但在覆盖人类知识多样性和长尾上有巨大缺口——因此在很多下游任务中还必须依赖人工标注数据进行有监督微调（Supervised Fine-Tuning）才能达到可用的水平。对比现在 ChatGPT 几乎不需要针对性微调即能完成纷繁复杂的大量不同任务——足见数据规模和参数量对「通用零样本能力」的解锁是根本性的。


## 第 58 页 · 小有所成——GPT-2 模型

**PPT 内容**：
- 虽然 GPT-1 锋芒未露，但 OpenAI 还是选择在 Decoder-only 路线上继续深耕，并在 2019 年 2 月发布了更为强大的 GPT-2 模型。

### 逐页讲解

GPT-2 是 Decoder-only 路线的一个重要概念验证——它在 GPT-1 的基础上大幅扩大了各个维度——参数量从 1.17 亿扩展到最高 15 亿（~12 倍）。训练数据从单一的 BookCorpus 升级为 40GB 的高质量 WebText 数据集。GPT-2 的最大惊喜是在**零样本**能力上的惊人飞跃——在没有任何针对任务的微调的情况下，模型仅通过自然语言提示就能独立完成翻译、摘要、问答等任务——虽然效果当时还远不如经过微调的专用 BERT，但这展示了自回归语言模型的一次本质性「能力涌现」。OpenAI 甚至因担心 GPT-2 生成高质假新闻的风险而当时只开源了最低参数版本——这也间接反向炒作了自回归模型的潜在大规模威胁。


## 第 59 页 · GPT-2 训练方式

**PPT 内容**：
- GPT-2 沿用了下一词预测任务，但提升了预训练数据的数量和质量。它采用了全新的 WebText 数据集——包含 40GB 经过筛选和清洗的网络文本。完成了预训练后，在某些任务上可以**不进行微调直接推理**——大大增加了处理下游任务时的灵活性。

### 逐页讲解

GPT-2 最重要的创新不在于架构（和 GPT-1 一致），而在于证明了**当自回归模型的数据量和参数量跨过一个阈值时，它开始自然展现出此前不可预测的多样能力**。WebText 的数据采集非常精妙——只选择那些 Reddit（网络社区论坛）上被高度赞同点赞的外部链接到的网页——相当于用人类社交平台上的「众包认可」自动提纯高质文本——无需人工标注。这使得训练数据的质量显著优于前代单纯的通用网页抓取。GPT-2 零样本推理能力宣告了「特定任务微调」式路径的终极衰落序幕。


## 第 60 页 · GPT-2 模型架构与四个版本

**PPT 内容**：
- GPT-2 在一代的基础上进一步加大了参数数量，并发布了四个版本——GPT-2 Small (12 解码模块/768 隐维/12 自注意力头/1.24 亿参数)、Medium (24/1024/16/3.55亿)、Large (36/1280/20/7.74亿)、XL (48/1600/25/15亿)。

### 逐页讲解

GPT-2 四版本是业界的第一次「多尺度模型组」探索——同一架构挂载不同层数和宽度→同步报告各版本的性能曲线→使得「Scaling 曲线」变得直观可见。递增梯度从 1.24亿→3.55亿→7.74亿→15亿——能力与非微调的零样本表现出现了明显的正比关系——这是后来 Scaling Law（规模法则）被量化的最初实验证据。


## 第 61 页 · 崭露头角——GPT-3 模型与规模化

**PPT 内容**：
- 为了进一步提升任务泛化能力，OpenAI 进一步提升了模型和预训练语料规模，于 2020 年 6 月推出了第三代模型 GPT-3。

### 逐页讲解

GPT-3 将参数与数据规模的扩展升到了重磅可量纲级别——最大版本 1750 亿参数。一年之间从 15 亿极限跳到 1750 亿——这几乎是所有算力所能支撑的当时最大参数模型。GPT-3 的出现让整个 AI 界清醒——这些不同规模的能力、大量的 Few-shot/Zero-shot 表现并非先前微调改进能赶上的。从这件起——LLM 赛道「不设上限、持续向上」的巨头竞争彻底开跑。


## 第 62 页 · GPT-3 模型架构——八组参数字级

**PPT 内容**：
- GPT-3 大幅增加了模型规模，参数量从 1.25 亿 (GPT-3 Small) 到最高 1750 亿。庞大的参数量使其能够捕获更加细微和复杂的语言模式，显著提升了文本生成能力和上下文学习能力。
- 各版本参数：Small 12/768/12/1.25亿, Medium 24/1024/16/3.5亿, Large 24/1536/16/7.62亿, XL 24/2048/24/13亿, 2.7B 32/2560/32/27亿, 6.7B 32/4096/32/67亿, 13B 40/5120/40/130亿, 175B 96/12288/99/1750亿。

### 逐页讲解

GPT-3 全参数谱系八个版本呈现出的惊人的规模化曲线——层数（12→96）、隐藏维（768→12288）、注意力头（12→99）——全方位指数级别增长。1750亿当时是世界最大——综合支撑了所有后续 LLM 路径的可行性。这个极大的宽度意味着模型拥有超大容量的记忆和极多额参数可以学复杂的模式——使得极少样本甚至零样本的指令理解成为可能。


## 第 63 页 · GPT-3 的上下文学习（In-Context Learning）

**PPT 内容**：
- GPT-3 模型涌现出了良好的上下文学习（In-Context Learning, ICL）能力——仅通过输入文本中的任务描述和少量示例，就能够快速适应不同的应用场景。
- Zero-shot——仅在给定任务的自然语言描述的情况下预测答案
- Few-shot——除了给定任务描述，还提供一些示例（如2-3个问答对）

### 逐页讲解

⭐ GPT-3 最大、最显著、最具革命性的能力——In-Context Learning (ICL)。不像之前所有的 NLP 模型需要在每个任务上做单独的微调训练，GPT-3 仅通过在输入 Prompt 中给出几个例子（Few-Shot）甚至只给任务描述（Zero-Shot），就能完成全新的任务——情感分类、翻译、常识问答、代码生成。这些任务无需任何梯度更新和梯度计算——模型的前向传播一路根据 Prompt 中的示范领会了任务格式然后对测试样本输出符合规则的内容。这既非「记忆」某特定任务——因为提供了全新设计且没有经过任何训练数据的特定任务——它都能做。这是通用能力的一次令人震動的涌现——自此 NLP 开启了从「微调范式」到「提示范式」的大迁徙。ICL 成为所有当代 Prompt 工程的基础理论。


## 第 64 页 · 蓄势待发——InstructGPT 等衍生模型

**PPT 内容**：
- 在 GPT-3 的基础上，OpenAI 推出了一系列衍生模型，包括专注于代码生成的 Codex、能够进行网页检索的 WebGPT 等。其中最具有启发意义的是具有良好指令跟随能力的 **InstructGPT** 模型。

### 逐页讲解

GPT-3 虽然强大的同时也有很强的「不受控性」——经常生成脱离原始指示的输出、或表现出不符合人类期望的行为。InstructGPT 是通向 ChatGPT 的关键一步——在 GPT-3 基础上加入了**从人类反馈中进行强化学习(RLHF)**——让模型在人类对多份回答进行偏好排序后学习什么是「好的人类期望回答」。InstructGPT 的指令跟随能力显著超越了纯 GPT-3——这也是从一个强大的但略危险的巨大预训练模型到可控可用的商业服务模型所迈出的最关键一步。


## 第 65 页 · RLHF（人类反馈强化学习）——让模型对齐

**PPT 内容**：
- InstructGPT 模型通过引入人类反馈强化学习（Reinforcement Learning from Human Feedback, RLHF），来提升模型对用户指令的响应能力。

### 逐页讲解

⭐ RLHF 是当代 LLM 最核心的后训练范式（第九章和本章第二板块有详细串联）。大致流程——预训练好后的模型每次针对一个问题生成多个回答，人类给回答打分和排序→训练一个奖励模型（Reward Model）来学习模拟人类偏好→最后用强化学习（PPO）根据奖励模型的反馈去微调基础语言模型以输出更符合人类期望的文本。

这解决了预训练 LLM 一个根本性的矛盾——预训练目标「最大化语料的概率」并不等于「生成人类喜欢的好用的回答」。RLHF 让模型的目标从统计似然转向了符合人类价值观和偏好。


## 第 66 页 · 一鸣惊人——ChatGPT 及 GPT-4 等模型

**PPT 内容**：
- OpenAI 随后推出了聊天机器人 ChatGPT（Chat Generative Pre-trained Transformer），标志着一种新的服务模式 **LLMaaS（LLM as a Service）** 的出现。
- 轻松调用——用户可以通过网页端或者 API 来轻松使用预训练后的 ChatGPT 模型，无需在本地进行模型部署。
- 逐渐闭源——自 ChatGPT 开始，GPT 系列模型走向闭源，OpenAI 不再公开模型及其参数配置，主要通过 API 形式来进行推理和微调。

### 逐页讲解

ChatGPT（2022 年 11 月发布）从技术角度看是 GPT-3.5 + InstructGPT 的 RLHF 对指令跟随的优化——但它在产品形态和商业模式上彻底改变了一切——**LLM 即服务（LLMaaS）**。普通人不需要任何 AI 技术知识——浏览器打开→打字聊天→获得回答——体验空前直接。ChatGPT 两个月内用户破亿——创下了消费互联网历史纪录——这比任何学术论文都能向下传达强大 NMT/LLM 的颠覆性能力。

闭源转向也自此起——GPT-3 以前的所有 GPT 模型均公开了模型架构乃至权重——但从 ChatGPT 及 GPT-4 起这些核心信息全部不公开——转变为纯 API 付费/订阅模式——OpenAI 的商业路径以模型服务为核心现金流——而不再以公开发表和社区共享为基础。这对后续所有大模型公司（Google、Claude、百度、字节等）的产品策略产生了深远影响。


## 第 67 页 · 大放异彩——GPT-4 模型

**PPT 内容**：
- 四个月后，OpenAI 继续发布了 GPT-4 模型。相较于 ChatGPT，GPT-4 能更好地理解复杂语境、生成连贯文本。此外，GPT-4 还引入了**对图文双模态的支持**，扩展了其在图像描述和视觉问答等领域的应用。

### 逐页讲解

GPT-4（2023 年 3 月）是新一轮技术联动的结果——它首次将大语言模型从纯文本+代码过渡到可以接收图像（图片输入）并基于图像内容进行理解的「多模态 LLM」——虽然图像端的输出能力受限（仍是文本输出），但可直接「看懂」一张图表——理解并分析数据、描述照片、阅读手写文字等。这为大语言模型的表达能打开了一个全新的维度——GPT-4 的复杂推理、长文理解、多步骤逻辑、数学证明能力也显著强于 GPT-3.5/3。


## 第 68 页 · GPT-4o——更快更全模态

**PPT 内容**：

### 逐页讲解

## 第 69 页 · LLaMA 系列模型——开源路线

**PPT 内容**：

### 逐页讲解

LLaMA 在 GPT 闭源后成为开源社区最重要的基座。LLaMA-1 在 2023 年 2 月公布时直接向学术界和研究机构提供权重文件（首次真正开源大体量级 LLM）。这一事件瞬间引爆了全球开源社区的二次创新——大量微调衍生物 (Alpaca/Vicuna/CodeLLaMA/LLaVA 等) 在短短数周内涌现——构建了一个全新的开源大模型生态链——让中小公司和个人开发者也能基于最高质量的基座模型做定制部署——对抗闭源巨头的垄断。


## 第 70 页 · LLaMA 与 GPT 的全参数对比

**PPT 内容**：
- GPT 系列的升级主线聚焦于模型规模与预训练语料的同步提升，而 LLaMA 则在模型规模上保持相对稳定，更专注于提升预训练数据的规模与质量。
- 对比表：GPT-1 (2018 1.17亿/5GB), GPT-2 (2019 15亿/40GB), GPT-3 (2020 1750亿/1TB); LLaMA-1 (2023 652亿/5TB), LLaMA-2 (2023 700亿/7TB), LLaMA-3(3.1) (2024 700亿~4050亿/50TB)。

### 逐页讲解

GPT 和 LLaMA 是两条发散的扩展曲线——GPT 从开始就在拼命扩参数（GPT-3 1750亿），LLaMA 最开始选择了较小的参数量（最大 652亿）但极大增强了训练数据量（5TB vs GPT-3 1TB）——遵从了 DeepMind Chinchilla 的最优配比原则（参数和数据几乎同等重要）。这个策略的结果就是——在同等参数量下 LLaMA 的性能远远超出当时所有同等模型（因为得到更充分的数据训练）——效率和性价比极高。LLaMA-3/3.1 将数据和参数同时推至 4050 亿参数和 50TB 时——已然达到替代 GPT-4 级别且全开源——开源社区的尖端水平正式与传统闭源巨头持平。


## 第 71 页 · LLaMA-1 模型

**PPT 内容**：
- 在 Chinchilla 扩展法则「小模型 + 大数据」的指引下，Meta AI 于 2023 年 2 月推出了最早版本的模型——旨在以大规模的优质数据训练相对较小的模型。

### 逐页讲解

LLaMA-1 的策略就是在业界疯狂追逐参数量的时刻逆势而行——坚持 Chinchilla 的最优配比原则：模型和数据同比例增加，而不是一昧加大参数。因此 LLaMA-1 的参数看起来不如同期的 GPT-3 庞大——但其效果凭借更充分的数据训练而显著超出同等参数量的其他模型。


## 第 72 页 · LLaMA-1 架构的关键改进点

**PPT 内容**：
- LLaMA-1 采用了与 GPT-1 同样的网络架构，但在原始词嵌入模块、注意力模块和全连接前馈模块上都进行了有效优化。
- 词嵌入模块——使用**旋转位置编码（RoPE）**替代绝对位置编码，提升对序列顺序的理解
- 注意力模块——借鉴 GPT-3 的 **Pre-Norm** 策略，对自注意力和前馈网络的**输入**进行正则化（而不是对输出做后归一化）
- 全连接前馈模块——将 Transformer 的 ReLU 激活函数替换为 **SwiGLU**

### 逐页讲解

这三项改进是 LLaMA 系列在技术上的核心创新，后续几乎所有开源 LLM 都沿用了同样的方案：

**RoPE（旋转位置编码）**：传统的绝对位置编码是给每个位置分配一个固定的位置向量，直接加到 Token 嵌入上。RoPE 则把位置信息编码成「旋转矩阵」，通过将 Query 和 Key 向量分别绕特定角度旋转来注入相对位置信息。RoPE 使得注意力分数天然地取决于两个 Token 之间**相对**位置的差异，而非它们各自的绝对位置——这让模型在遇到比训练时更长的序列时泛化性更佳。

**Pre-Norm**：常规的 Transformer 在每个子层的**输出**之后做层正则化（Post-Norm）。Pre-Norm 反其道而行——在每个子层的**输入**之前做层正则化。实践证明 Pre-Norm 对深层 Transformer 的训练稳定性有极其显著的提升——梯度在深层网络中的传播更加稳定平滑——不需要 Warmup 等高级训练技巧也能轻松训练收敛。

**SwiGLU**：ReLU 是非光滑的 (在 0 点不连续)。SwiGLU 结合了 Swish 的平滑非线性和 GLU 的门控机制，对于前馈网络的隐层表达力有显著的增强——同等参数量下能获取更丰富的特征交互。


## 第 73 页 · LLaMA-1 四个版本参数

**PPT 内容**：
- LLaMA-1 发布了四个版本——7B (32 解码模块/4096 隐藏维/32 自注意力头/67亿参数), 13B (40/5120/40/130亿), 32B (60/6656/52/325亿), 65B (80/8192/64/652亿)。

### 逐页讲解

LLaMA-1 的四个版本覆盖了从低到高不同资源规模的部署需求——使得开发者可以根据自身硬件条件灵活选择最适宜的版本（7B 可以在单张消费级 GPU 上运行推理；65B 则需要多卡或 A100/H100 类高显存专业卡）。这一灵活性极大促进了开源社区中的衍生和各种垂直领域微调模型的繁荣。


## 第 74 页 · LLaMA-2 模型

**PPT 内容**：
- 五个月后，2023 年 7 月 Meta AI 进一步优化和扩充了训练数据，发布了第二代模型 LLaMA-2。除了数据量的扩大，LLaMA-2 还引入了 RLHF 训练范式，并在其基础上结合了拒绝采样来进一步提升模型性能。

### 逐页讲解

LLaMA-2 的最大升级在于「对齐」——加入了类 InstructGPT 的 RLHF 训练——让开源模型首次具备了与闭源对话模型（ChatGPT）可比的指令跟随和安全性。同时还采用拒绝采样（自动评判生成质量，差的不采样进训练）提升了样本级内容质量和回答的可靠度。自此——LLaMA 不仅参数强——指令跟随和安全对齐也上去了——开源模型正式对齐商业产品的交互体验。


## 第 75 页 · LLaMA-2 架构——引入 GQA

**PPT 内容**：
- 在 LLaMA-1 的基础上，LLaMA-2 还额外引入了分组查询注意力（Grouped Query Attention, GQA）来提升计算效率，并最终发布了四个版本（7B/13B/34B/70B）。

### 逐页讲解

GQA——将多个 Query（查询）头合并共享到更少的 Key/Value（键/值）组的注意力执行加速技巧。大型模型的推理瓶颈之一是 KV-cache（键值缓存）的高显存占用——GQA 使得 KV 缓存的数量减少数倍，同时不显著牺牲模型质量，对大规模推理和长文本回答的部署效率极为有利。


## 第 76 页 · LLaMA-3/3.1 模型——指数飞跃

**PPT 内容**：

## 第 77 页 · LLaMA-3.1 405B 的规模感知

**PPT 内容**：
- 在 LLaMA-3.1 405B 的参数量、训练数据量及所需算力的直观对比展示。
- 参数量：405B（四千亿参数）——对比 2016 年击败李世石的围棋程序 AlphaGo 的参数量仅为 0.065B——LLaMA-3.1 405B 相当于 6000 多个 AlphaGo
- 数据量：训练数据高达 15 万亿 Token——四库全书约 3 亿 token →其数据量约等于 50000 套四库全书
- 计算量：训练所需计算量高达 3930 万个 GPU 小时（Nvidia H100）。单一 H100 一张 GPU 相当于数百台个人电脑的单一算力——训练所需如一台个人电脑连续运行超过 50 万年

### 逐页讲解

⭐ 这组超巨大的数对比是为了让学生有一个直观的「体感」——大语言模型到底大到了什么程度。

AlphaGo 在 2016 年因其惊人的围棋能力令全世界哗然——它的参数量（逼近650万）在当时已然巨大无边。LLaMA-3.1 405B 的参数是 AlphaGo 的约 6200 倍。知识覆盖范围——50000 个四库全书的内容量 ——十五万亿 token 不可能被任何一个人类阅读完毕——大模型把人类所有可获取的文字信息都蒸馏压缩到了 4050 亿个参数中。

3930 万个 H100 GPU 小时的训练成本换算成金钱——H100 在云端的时租约为 3-5 美元/卡/小时 ——总训练成本约数亿美元——而单靠个人电脑需要连续跑 50 万多。这就是当代大模型产业「只能用顶级互联网公司的基础设施才能铸造」的根本原因。


## 第 78 页 · LLaMA 衍生模型生态

**PPT 内容**：
- LLaMA 模型的开源共享吸引了众多研究者在其基础上继续创作。众智众创为 LLaMA 营造出了一个多样的、充满活力的研究生态。
- 性能改进类（持续微调使通用任务性能更强如 Alpaca/Vicuna）、垂域任务类（专用域微调如 CodeLLaMA/FinLLaMA（金融/代码））、多模态任务类（整合视觉编码器扩展到跨模态任务——LLaVA/MiniGPT4）。

### 逐页讲解

LLaMA 的开源激发了一个庞大的二次创新网络——以极短的时间形成了整套多个垂直解决方案。Alpaca——在 LLaMA-7B 基础之上用 GPT-3.5 生成的指令数据集进行自我蒸馏——50 美金的微调成本即可复现类 ChatGPT 的指令跟随能力——揭示了基础模型能力的惊人可迁移性。CodeLLaMA 和 FinLLaMA 等垂直模型则在基础 LLaMA 框架上往编程/金融类数据定向微调——获得了远超通用模型的专用能力。LLaVA 等则将视觉编码器整合到 LLaMA Decoder 内部——使 LLaMA 获得图像理解+生成能力——成为多模态 LLaMA。LLaMA 生态在半年内从单纯的「一个语言模型」成长为「开源智能综合体系的组件」——这是 OpenAI 闭源生态完全不同的另一条路。


## 第 79 页 · Decoder-only 架构总结

**PPT 内容**：
- 汇总 GPT 和 LLaMA 全系列——GPT-1(2018 1.17亿/5GB), GPT-2(2019 1.24-15亿/40GB), GPT-3(2020 1.25-1750亿/1TB), ChatGPT(2022/未公开), GPT-4(2023), GPT-4o(2024); LLaMA-1(2023 67-652亿/5TB), LLaMA-2(2023 70-700亿/7TB), LLaMA-3(3.1)(2024 80-4050亿/50TB)。

### 逐页讲解

Decoder-only 两个旗舰系列各自走出了不同的技术-商业战略路线——闭源超级规模路线 vs 开源效率数据饱和路线。到2026年——两者在多个基准测试中已极相近——「开源追上闭源」的关键转折已经到来。这场两端巨头对决仍是未来几年大语言模型的中心叙事。

# 二、大语言模型的后训练（第 81-95 页）


## 第 81 页 · 会接龙不等于会说话

**PPT 内容**：只接受过接龙训练的大模型容易产生「复读机效应」。示例——问孙悟空得了肾结石算不算怀孕，未经历 RLHF 的基础模型给出大量复读机式的重复/跑题/不当回答。

### 逐页讲解

预训练语言模型本质上是文本生成串接——一问之下最自然的自回归接续是模仿训练数据中类似的出现模式——不一定是对提问意图的理解。这会导致它在问答格式下生成一串不适当或直接贴标签的荒唐回答（复读机现象）。因此必须在预训练基础上加一道「教它回答格式和内容安全性」的工序。


## 第 82 页 · 指令微调——让模型「会说话」

**PPT 内容**：可以通过指令微调让模型「会说话」。指令微调构造「说话秘籍」（高质量的问答对数据集），然后对大模型进行监督学习微调。示例——同一个问题，指令微调后的模型能给出符合语境、信息性的回答。

### 逐页讲解

指令微调（Instruction Fine-Tuning / SFT）是后训练的第一步。收集大量人类写好或由更高级模型生成的（问题，高质量回答）对，把这些优质答案作为监督信号→用交叉熵损失对预训练基座模型做一次全量或部分参数的微调。这步之后——模型学会了「问答的格式」和「什么是好回答的基本范式」——不再是乱接龙——而是回答问题。


## 第 83 页 · 会说话不等于说好话

**PPT 内容**：「会说话」不等于「说好话」——指令微调后的大模型仍可能不符合人类意图或价值观。给出一个示例——某个模型对同一问题给出攻击性、不当、用语不雅的回复（大量人身攻击内容）。

### 逐页讲解

指令微调教会了模型对话形式——但并没教会它**什么是好、什么是坏、什么有害、什么被禁止**。模型可能在对话中对用户的提问做出攻击性或传播不良信息的输出——这甚至在格式上看似「会说话」但价值观与安全约束缺失。这意味着还需要额外的「价值对齐」——确保模型输出符合诚实（Honest）、无害（Harmless）、有用（Helpful）的 3H 原则。


## 第 84 页 · 不说好话可能产生的风险

**PPT 内容**：不说「好话」可能产生严重的社会伦理风险甚至带来 X-Risk。需要大模型与人类价值观进行对齐——3H 原则（诚实、无害、有用）。

### 逐页讲解

大语言模型如果输出涵盖种族歧视/人身攻击/教导违法操作/煽动暴力等不安全信息将对使用者和社群构成极大的伤害——在极端情况甚至被列为潜在的「存在风险(X-Risk)」。因此业界在对齐研究中设立了三项关键原则——诚实、无害、有用——作为所有进一步 RLHF 对齐的目标标准。对齐目标是将语言模型的行为从不安全、不道德区域导向安全、建设性的对话方向。


## 第 85 页 · 对齐第一步——训练奖励模型

**PPT 内容**：对齐的第一步：人类对不同答案给出标注，训练奖励模型——蒸馏人类价值观打造「价值判官」。针对同一问题，人类对三个不同回答分别打分（如 1 分、6 分、9 分）以标注偏好。然后训练奖励模型（Reward Model）去预测人类对任何回答的偏好分数。

### 逐页讲解

三个不同回答的评分例子：
- 回答1「算个鬼啊！你这脑洞比孙悟空的金箍棒还能乱伸……（攻击性）」→1 分
- 回答2「算！孙悟空本就是从石头里蹦出来的石胎孕育……（创意但违背常识）」→6 分
- 回答3「从不同角度有不同答案：从娱乐神话角度可以算特殊类型的创意，从常识严谨角度不算怀孕……」（客观、全面、理性）→9 分

这个评分过程给奖励模型提供了监督信号——让它学会从众多回答中识别人类偏好的好坏排序。奖励模型本质上是一个评分函数——日后在 RL 中将指导策略模型。


## 第 86 页 · 对齐第二步——强化学习注入人类偏好

**PPT 内容**：对齐第二步：通过强化学习（PPO）将人类价值偏好注入语言模型。过程：待对齐模型不断生成回答 → 批判模型（Reward Model，固定不变）根据人类偏好对生成文本打分 → 用 PPO 算法将奖励反馈更新入待对齐模型的参数（不断提升其获取高偏好评分的倾向）。

### 逐页讲解

⭐ RLHF + PPO 的完整操作流程——首先在预训练 + SFT 之后的基座模型上冻结 Reward Model（训练好的评分器），然后在其下用 PPO 强化学习持续微调解码层参数使模型最大化生成出被奖励模型认为「好」的输出。与此同时，PPO 中包含一个 KL 散度惩罚项——确保模型在优化时不会跑偏太远离原始 SFT 的对话策略（否则可能出现奖励过拟合——生成某种极端但偶然高评分的模式语言——而非真正保持语义自然性）。

训练后——模型输出既保持了指令跟随的格式——也内化了人类的价值审美和安全标准。


## 第 87-90 页 · 思维过程的两个系统 & 思维链 CoT 推理

**PPT 内容**：思维分为 System-1（直觉快、自动化）和 System-2（理性慢、逻辑推理）。思维链 CoT——诱导模型一步步输出推理过程（「Let's think step by step」），将 System-2 的逻辑链条显式展开在作答中→在数学、逻辑与常识推理等任务上显著提升准确性。若要更深提升还需给 CoT 参考示例、或把 System-2 思维内化到模型参数中（通过强化学习刻进骨子里）。

### 逐页讲解

System-1 / System-2 的区分源自丹尼尔·卡尼曼的行为经济学研究——直觉快思考（System-1）和慢逻辑推理（System-2）构成人类两种思维，大模型单纯自回归可视为类 System-1 模式（直接产生答案、缺乏推敲）。思维链提示（CoT）在输入提示中加入「请一步步地思考」强制模型在输出最终答案前先输出中间推理步骤→等于将 System-2 引入回答流程——每步都可自我监督和检查→减少跳跃错误。当模型自行推理能力依然不足时——提供参考的推理示例（Few-shot CoT）引导模型模仿正确的推理逻辑结构→辅助内化推理密度。


## 第 91-95 页 · DeepSeek R1 与 GRPO — 推理能力爆发

**PPT 内容**：强化学习将被诱导出的最大量的思考过程刻进模型内部。DeepSeek R1 使用 GRPO（Group Relative Policy Optimization）方法——不做批判模型，直接「干中学」——用同一问题的多个平行答案相对排序作为奖励→训练策略模型→最终以极低成本获得世界一流推理能力并在训练过程中自主涌现「Aha Moment」——模型学会在强化过程中暂停重审思路。

### 逐页讲解

GRPO 是 DeepSeek R1 对 RLHF 的重大创新——传统 PPO 需要额外维护一个昂贵的奖励模型并频繁进行评分调用——GRPO 不需要——为同一道问题生成多个回答，根据其正确性与是否有思考过程（<think>标签内的内容）自动给出相对奖励排序——并用组内相对优势加策略择优更新（替代绝对奖励模型的反馈）。这使得 RL 训练流程简化且计算成本极大幅度削减——DeepSeek R1 以 OpenAI 1/18 的训练成本和 1/10 的团队规模——在数学、编程和逻辑类任务上达到了与其持平甚至超出的一线水准。

更惊人的是「Aha Moment」——在漫长的 GRPO 训练中——模型在无人工提示下学会了暂停当前不准确推理——自发地去重新检查中间步骤和逻辑——自行从漂移了的方向再调整回来输出更优答案——这种自我校正（self-correction）是推理高级涌现的极典型标志。


# 三、参数高效微调（第 96-156 页）


## 第 96-97 页 · 全量微调的挑战

**PPT 内容**：全量监督微调需要更新所有模型参数——当面对大模型时消耗大量存储和计算。GPU 内存不足（7B 模型全微调需约 60GB 显存，消费显卡无法满足）、全量微调效率低。

### 逐页讲解

全量监督微调意味着所有预训练权重、所有梯度、所有优化器状态要在前向传播和反向传播间全周期存在 GPU 显存中——举例 7B 的 LLaMA 全微调占用约 60GB——远超 24GB RTX 4090 等单卡上限。因此没有云集群的普通用户无法直接微调大模型——这就需要参数高效微调技术（仅更新少量额外参数或选择部分参数更新→大幅降低显存和存储需求）。


## 第 98-99 页 · PEFT 定义与三大优势

**PPT 内容**：参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）——避免更新全部参数，在保证微调性能的同时减少更新的参数量和计算开销→三大优势：计算效率（仅少量参数需要梯度计算→显存和算力开销低）、存储效率（仅存少量参数调整值→每个任务仅需几 MB）、适应性强（快速适配不同任务且性能不损失）。

### 逐页讲解

从全量到 PEFT 的转变本质上是从「对所有参数都重新对齐」到「仅对关键偏置参数做插入式适配」。三大优势分别解决了 GPU 显存大、硬盘空间浪费、以及多任务时需保存巨大独立的完整模型副本等困扰。特别是存储——每个下游任务只需一个几 MB 的额外权重文件（LoRA 插件）即可动态载入——使得同一模型可同时服务成百上千个下游任务而存储成本极低。


## 第 100 页 · 显存对比——全量微调 vs PEFT

**PPT 内容**：7B 模型全量微调需要 60GB（BF16），LoRA 仅需 16GB；13B 全量 120GB vs PEFT 32GB；30B 全量 300GB vs PEFT 64GB。数据来源：LLaMA-Factory。

### 逐页讲解

全量微调显存随着参数规模线性增长——7B 要 60GB、13B 要 120GB、30B 要 300GB——这已超出几乎所有民用级别的显卡和大多数中小服务器的承载极限。PEFT（如 LoRA）则分别仅需 16GB / 32GB / 64GB——大幅拉低了微调门槛——让一般开发者在消费级硬件上（如单 RTX 4090 24GB 可微调 7B、甚至 13B 的下调 LoRA 版本），才能真正落地大模型定制应用。


## 第 101 页 · 存储效率——PEFT 可显著节省存储

**PPT 内容**：PEFT 仅需保留部分微调参数——显著降低微调模型的存储空间——适合存储受限设备。LLaMA-3 13B (FP16) LoRA rank=32 的演示——全量微调 26GB vs LoRA 适配器仅几 MB。

### 逐页讲解

全量微调模型结果 = 完整 13B 参数的一个新权重文件（26GB）；PEFT = 将原模型权重冻结 + 极小的低秩矩阵存储更改（几 MB）。因此面对数百个不同微调场景（法律、医学、编程、教育……）全量微调需要上百份完整参数的 TB 级硬盘占用，PEFT 同等数量仅需几十 MB 到几百 MB。


## 第 102 页 · 适应性强——PEFT 不会牺牲性能

**PPT 内容**：PEFT 在降低微调成本的同时，保证微调性能不受影响。T5 实验——LoRA 在所有数据集上都匹配甚至超越了全量微调的基线。GPT-3 实验——LoRA 在所有数据集上也达到了同等或更优的性能表现。

### 逐页讲解

PEFT 经常被质疑「只更新少量参数是否性能不够」。大量实验数据证实——即便只更新少于 1% 的参数（如 LoRA），微调后的性能在整个秩调优合理的前提下可以与全微调打平甚至在某些小样本条件下反超（因为少量参数降低了过拟合风险）。这使得 PEFT 成为业界微调大模型的标准方案——并非妥协手段——而确实是同等的、甚至更优的选择。


## 第 103-105 页 · PEFT 三大分类

**PPT 内容**：主流 PEFT 方法分三类——参数附加（增加新参数）、参数选择（选原模型部分参数更新）、低秩适配（LoRA——用低秩矩阵近似权重更新）。

- 参数附加按插入位置分三类——加在输入（Prompt-tuning 等；最柔、不改变模型内部）、加在模型（Adapter/Prefix-tuning；各层插入瓶颈模块）、加在输出（Proxy-tuning 等解码端约束）
- 参数选择——BitFit 只更新偏置，Child-tuning 学习选择参数子集
- 低秩适配——LoRA 系列（低秩矩阵近似全权重更新）


## 第 106-110 页 · Prompt-tuning——参数附加在输入

**PPT 内容**：额外参数附加到输入嵌入中。引入软提示（Soft Prompt）——可训练的连续向量——在模型输入时前置在真实 Token 嵌入之前→微调时只有软提示参数被更新而模型其余全部冻结。软提示长度和初始化方法对效果有显著影响——模型规模增大、最优提示长度可更短；使用分类头的采样嵌入来初始化软提示更佳。

### 逐页讲解

Prompt-tuning 是 PEFT 中侵入性最低的——完全不需要查看或修改模型内部结构——只要在输入前面加几段可训练向量。软提示通过反向传播从下游任务标签接收梯度，逐步调整自身参数以引导 LLM 的输出偏好（类似在模型眼前放置一个训练过的透镜）。不同于手工编写的硬提示（离散 Token 固定不变）——软提示在学习上是连续的、可被精确梯度优化的——灵活性更高。缺点是不可读（学到的是连续向量无人类解读形式）、对短软提示需要大模型才有效。


## 第 111-115 页 · Adapter-tuning——参数附加在模型中

**PPT 内容**：加在模型的方式在预训练模型的每层 Transformer Block 后插入小型可训练的适配器模块（Adapter）。适配器结构——瓶颈形状：下投影将 d 维压缩到低维 r→非线性激活→上投影恢复回 d 维。总参数量 2dr + d + r，r ≪ d。全量冻结原模型仅对 Adapter 和最后的分类头进行微调→大幅缩减训练参数量并维持预训练知识。优势——参数效率极高、可不同任务级联多 Adapter、预训练的原始知识不受损。

### 逐页讲解

Adapter 设计核心就是每个模块插入的信息压缩瓶——先将输入的 768 维高维向量压缩到一个低维瓶颈空间（如 48 维）中处理再扩回原维输出——瓶颈压缩使参数数大幅缩小 (d×r vs d×d 的高矩阵)→同时对下游任务的关键信号进行了有效的提取。在下游对每个任务可一一插入和保存各自的 Adapter——从而不需存储整个模型的独立微调副本——高效且完整保留原语言模型的所有常识记忆。


## 第 116-121 页 · 代理微调（Proxy-tuning）——参数附加在输出

**PPT 内容**：针对无法直接获取大模型权重的黑盒模型（闭源 API）和大模型过重无法直接全量/任何内部微调的场景。解——训练一个轻量反专家模型 + 专家模型——在每个生成时间步计算专家与反专家模型对下一 token logits 的差异——将此差异叠加到目标大模型的输出 logits 分布上→再通过 softmax 采样——实现解码时约束对大型黑盒模型的引导。

### 逐页讲解

Proxy-tuning 本质上是「动态调整大模型的下一个词候选偏序」。不需要修改大模型内部参数甚至无需其代码——只需要在大模型生成每个 token 的 logits 分布上，加上从小模型学来的偏好偏移信号（专家喜欢-反专家不喜欢），然后从这个调整后的分布中采样。这使得我们可微调一个小模型（成本极低）即可引导比其大几个数量级的巨大闭源模型输出偏好方向。


## 第 122-130 页 · 参数选择方法 ——BitFit 与 Child-tuning

**PPT 内容**：BitFit——仅更新神经网络中所有层级的偏置项 (Bias) 和最终分类头→偏置项仅占所有参数的约 0.08-0.09%，但能在 GLUE 多项评测上与全量微调相当同时训练稳定性更高。Child-tuning —— 在反向传播中用梯度掩码屏蔽掉一部分子网络（仅对被选中的子网络参数更新）→从而通过选择子网络最终实现选择性参数更新。

### 逐页讲解

BitFit 的发现特别反直觉——仅更新那占比微乎其微的偏置项参数就能匹敌全微调效果，这说明对于许多下游任务大模型的核心知识并不需要重新学——只需要调整每层的输出基线（bias shift）即可对输出分布做全局偏置控制而无需扭曲所有权重。Child-tuning 像在训练时随机丢掷节点——形成子网络选择——能「保留部分预训练优良知识」不被精调破坏。


## 第 131-143 页 · LoRA —— 低秩适配核心

**PPT 内容**：本征维度假设——微调时仅需要在一个非常低秩的子空间中进行更新即可达到全参数更新的近优效果。LoRA 的公式：ΔW = B × A（B: d×r, A: r×d, r ≪ d）。W'=W+ΔW，冻结原 W，只训练 A 和 B。初始化——B 全部为零、A 以随机高斯初化→确保初始 ΔW=0 不破坏预训练权重。推理时 LoRA 适配器可无缝合并回原权重无额外推理开销。秩的选择 (r) 的影响——简单任务低秩 (1~8) 足够、复杂任务宜高秩 (32~64)；施加不同类型矩阵 (Q,K,V,O) 效果好于单一矩阵。分多项实验、存储和适配生态相关的详细讨论了 LoRA 各种变体并提及其在 GPT-3、LLaMA 上的微调实测。

QLoRA——将原模型参数 4-bit 量化存贮节省 ~60% 显存；S-LoRA——多 LoRA 同时推理服务场景的统一内存池管理+预取调度优化 GPU 利用率。

### 逐页讲解

LoRA 是整章 PEFT 板块最核心的技术，所有最近的参数高效微调工具和平台（LLaMA-Factory, Axolotl, HuggingFace PEFT 库）都以 LoRA 为默认和首推工具。LoRA 的训练费用极低（适配器的参数量通常仅原模型的 0.1~1%），微调后可把适配器单文件（几 MB）拷贝分发→无门槛布署（直接调用基础模型 + 挂载对应 LoRA 文件）。多 LoRA 插件组合甚至可实现不同任务间的知识迁移（例如用组合金融分析 LoRA + 代码 LoRA 来生成金融编程项目）。


## 第 144-156 页 · 全章总结与变体纵览
