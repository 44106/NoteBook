### 逐页讲解

这是全课程的最后一章。前十一章已经从计算机视觉（CNN、检测分割、生成模型）一路走到自然语言处理（表征学习、语言模型、大语言模型架构）。这一章要解决一个非常现实的问题：**大语言模型虽然强大，但它会「胡说八道」。** 这就是所谓的「幻觉」（Hallucination）——模型生成的文字表面上流畅可信，但事实内容与真实世界不符。

本章的主题「检索增强生成」（Retrieval-Augmented Generation，简称 RAG）是目前缓解幻觉最有效、最广泛落地的技术方案。它的思想非常朴素——模仿人类的行为：当我们被问到不确定的问题时，会先查一下搜索引擎或翻一下参考书，然后再回答。RAG 就是让大语言模型在生成答案之前先去外部知识库「查资料」，用查到的事实来约束和辅助后续的生成。

本章 114 页的内容分为五个板块：RAG 的架构设计（黑盒 vs 白盒）、知识检索模块的完整技术栈（知识库构建、查询增强、各类检索器、重排序）、生成增强策略（何时增强、何处增强、多次增强）、降本增效技术、以及 RAG 在企业中的落地现状。每一个板块都是 RAG 工程实践中不可或废的组成部分。


## 第 2 页 · 幻觉现象——九个主流模型一场大考

**PPT 内容**：
- 大语言模型生成的内容可能存在「幻觉」现象：生成内容看似合理但实际上逻辑混乱或与事实相悖。
- 六个国内外主流大模型 / 助手（GPT-4o / Claude 3.5 / Gemini / 通义千问 / 文心一言 / 豆包 / Kimi）被问同一个问题：「9.11 和 9.9 哪个大？」——部分模型回答错误（将 9.9 判断为大于 9.11）。

### 逐页讲解

这是本章极其有力的一个开场。PPT 展示了当前国内外最顶尖的多个大语言模型或助手在面对一个极其简单的数值比较问题——「9.11 和 9.9 哪个大？」——竟然出现了不同程度的错误。有些模型把 9.11 和 9.9 理解成了软件版本号（在这种语境下 9.11 确实比 9.9 大），而非小数数值比较。也有一些模型的 Token 化方式将「9.11」和「9.9」分别拆成了奇怪的子词组合（如「9」「.」「11」和「9」「.」「9」），导致模型在自注意力聚合时无法正确建立「这些 Token 构成一个完整小数」的认知。

这个例子传递了几个重要信息：

第一，**幻觉不是低端模型的专属问题**——GPT-4o 和 Claude 3.5 这种当前最顶尖的模型同样可能出错。幻觉是大语言模型结构性统计本质的固有缺陷，而非某个具体模型的工程质量问题。

第二，**幻觉不只是「事实性知识错误」**——它也包括逻辑推理错误、数值计算错误、常识判断错误。这些错误的根源在于大语言模型本质上是一个「统计预测器」——它学习的是「在类似上下文中，下一个 Token 最可能是什么」，而不是「这个事实是否正确」或「这个推理是否逻辑严密」。

第三，**Token 化过程本身就可能引入系统性偏差**——数字、日期、公式在 Token 化时可能被切分得支离破碎，导致模型在聚合时需要跨多个 Token 去拼凑完整信息，而中间任何一个环节的统计偏差都可能导致最终判断失误。

这个话题直接引出了本章的核心使命——如何让大语言模型在面临事实性问题时不再仅依赖自身不那么可靠的参数记忆，而是去外部查证事实依据。


## 第 3 页 · 幻觉的两大来源——训练数据 vs 模型本身

**PPT 内容**：
- 「幻觉」现象可能直接来源于训练数据中的知识错误，也可能源于训练出的模型本身对知识的掌握不足。
- 训练数据方面的来源：知识过时、知识边界、知识偏差
- 模型本身方面的来源：知识长尾、曝光偏差、解码偏差、对齐不当

### 逐页讲解

⭐ 这一页系统化地将幻觉的七大来源梳理为两条主线——数据侧和模型侧。这是一个非常实用的分析框架——帮你定位任何一个具体的幻觉案例的根因。

**训练数据侧的三个来源**：

**知识过时**：大语言模型的内部知识在训练完成那一刻就被「冻结」了。GPT-4 的训练数据截止于 2023 年某个时间点，此后世界上发生的一切事件——新的科学发现、政治变动、体育赛事结果——模型一概不知。如果问它 2024 年奥运会的金牌榜，它要么拒绝回答，要么自信地编造一个看似合理的答案。

**知识边界**：没有任何训练数据集能覆盖人类知识的全部范围。考拉基因组的具体基因数量、某个冷门小镇的历史事件、某位不太知名科学家的生平细节——这些信息如果在训练数据中本就不存在或极其稀缺，模型自然无法准确回答。

**知识偏差**：训练数据（主要是互联网文本）包含大量人类偏见、过时观点和事实性错误。如果训练数据中有 90% 的企业家描述都默认使用男性代词，模型就会学到这种偏差并在生成时将这种偏见传递给用户。

**模型本身的四个来源**：

**知识长尾**：即使某个事实在训练数据中存在过，如果它只出现了极少的次数（位于长尾分布的尾部），模型也很难牢固记忆。这就像你在备考时把某个生僻知识点只快速扫了一遍——考试时大概率想不起来。

**曝光偏差**：这个在第十章详细讲过——训练时因为 Teacher Forcing 模型接收的是正确的上文，但推理时接收的是自己生成的可能有错误的上文。这种训练和推理的分布差异会导致模型在自推链中逐步偏离正确的语义方向。

**解码偏差**：采样过程中的随机性（Top-K、Top-P 引入的随机采样）可能偶然选到低概率的不合理 Token，而这一步偏差被自回归机制逐级放大。

**对齐不当**：RLHF 阶段如果人类标注员的偏好数据本身存在问题（比如偏好冗长的回答而忽略事实准确性），对齐后的模型可能更擅长「说得让人类喜欢」而非「说得正确」。

这七个来源在现实中常常叠加作用。RAG 的核心思想就是绕过这些内部缺陷——不在模型参数的「黑盒记忆」里挣扎，而是直接去外部数据库查最新、最权威的事实。


## 第 4 页 · 幻觉案例——知识过时

**PPT 内容**：
- 知识过时：由于训练数据中的时间滞后，其中的知识可能在模型训练后又发生了更新，导致模型内部知识过时。
- 例子：某模型（DeepSeek-V2.5）在回答关于「西班牙队」的问题时，仍认为西班牙队是冠军，但事实是在模型训练截止后，新一届世界杯 / 欧洲杯的结果已经发生了变化。

### 逐页讲解

这个案例非常典型。假设 DeepSeek-V2.5 的训练数据截止到 2024 年初，而 2024 年 7 月的欧洲杯决赛在模型训练后才进行。当用户问「谁是欧洲杯冠军？」时，模型内部仍然「记得」训练数据中的上一届冠军，而不知道新结果。

知识过时带来一个根本性矛盾：重新预训练模型以更新知识成本极高（需要数千 GPU 运行数周甚至数月，耗资数百万至上千万美元），所以不可能频繁更新模型参数中的知识。RAG 的解决思路就是解耦「知识」与「参数」——知识放在外部数据库（可以实时更新），模型参数保持稳定，每次提问时即时从外部数据库检索最新信息。这样知识库可以在几秒钟内更新（比如新闻事件发生后立刻更新数据库），而模型本身不需要重新训练。


## 第 5 页 · 幻觉案例——知识边界

**PPT 内容**：
- 知识边界：由于训练数据的有限性，无法覆盖所有范围，且知识在训练数据采集完成后仍会不断新增。
- 例子：正确答案是「考拉的基因组包含了大约 26,000 个基因」——但 GPT-3.5 无法回答此类低频、高度专业化的冷门知识。

### 逐页讲解

考拉基因组的具体基因数量是一个非常低频、高度专业化的知识。即使在互联网的海量文本中，这个信息的出现次数可能也非常有限——可能只在几篇学术论文、少数科普文章中被提及。对于 GPT-3.5 来说，这个知识点在训练数据中的「曝光度」不足以让模型牢固记忆。

这揭示了 LLM 的一个天然局限——**并不是所有人类知识都在模型中有了等权重的表示。高频知识（如常识、流行文化）被模型记忆得比较准确，而低频专业知识（长尾知识）则可能记忆模糊甚至完全缺失。** 而在专业应用场景（医疗、法律、科学研究）中，恰恰是这些低频专业知识最为重要。

RAG 为这一问题提供了优雅的解决方案——专业领域的知识可以被系统地整理到外部知识库（如医学论文数据库、法律条文库、基因数据库），RAG 系统在回答时针对性地检索这些专业源，弥补 LLM 自身对低频知识的记忆缺口。这样就不需要依赖模型的「长尾记忆」来覆盖所有专业知识。


## 第 6 页 · 幻觉案例——知识偏差

**PPT 内容**：
- 知识偏差：训练数据中可能包含不实与偏见信息。
- 例子：GPT-4o 在描述「企业家」或其他职业身份时，默认以男性代词指代，而忽略女性身份的成功企业家——这反映的是训练数据中长期存在的性别偏见被模型内化了。

### 逐页讲解

知识偏差与大语言模型的社会伦理影响问题密切相关。这个例子展示的是模型在生成描述性内容时——用户并未指明企业家性别——模型自动使用了「他」这一男性代词，在统计上反映了训练语料（包括大量互联网文章、新闻报导、历史文献）中长期存在的性别角色定式。这些数据中的刻板印象——男性主导商业、女性主导家庭——在模型的无监督自回归训练中会被不加辨别地学到。

RAG 在缓解知识偏差方面能发挥作用——通过从多样化的、经过去偏处理的权威来源检索信息，为模型提供更平衡的上下文参照，减少训练语料中单一群体偏见对生成输出的主导。比如可以从学术数据库中检索到「女性企业家在各个行业都有杰出代表」的相关事实文档，作为 Prompt 上下文辅助生成。


## 第 7 页 · 幻觉案例——对齐不当

**PPT 内容**：
- 对齐不当：在模型与人类偏好对齐阶段中，偏好数据标注不当可能引入了不良偏好。
- 图解：用户问题 → 大模型 → 多个候选答案 → 人类标注进行偏好排序。如果偏好数据标注过程本身引入了偏差（比如给某些不符合事实但流畅华丽的回答打了高分），后续 RLHF 对齐后模型就会被带向不良偏好。

### 逐页讲解

RLHF 对齐的流程是：为同一个问题生成多个不同风格或方向的回答 → 人类标注员对这些回答进行偏好排序（哪个更好、更符合要求）→ 训练奖励模型学习人类偏好 → 用 PPO 根据奖励模型引导模型向高分偏好方向微调。

但如果标注数据本身存在问题——比如标注员更偏好「看起来自信果断」的回答而非「实事求是、承认不确定性」的回答——RLHF 对齐后的模型就会更倾向于在不确定时「自信地胡说」而非谨慎地表达不确定性。这就是对齐不当可能成为幻觉来源的原因。

RAG 在这方面提供了一个结构性的缓解策略：因为模型在回答时有外部文档提供的明确事实依据，它更不容易迷失在「过度自信但错误」的统计模式中。外部文档本质上提供了一种「硬约束」——模型在 Prompt 中清楚地看到事实数据，就不太会去凭空编造。


## 第 8 页 · 幻觉案例——知识长尾（模型侧）

**PPT 内容**：
- 知识长尾：训练数据中部分信息的出现频率极低，导致模型对这些知识的学习程度较低。
- 例子：正确答案是「美国心理学家 G.W.奥尔波特」——但 GPT-3.5 错误地回答为「卡尔·罗杰斯」。奥尔波特虽然是心理学史上重要的人物（人格心理学之父），但在大众互联网文本中的提及频率远低于更广为人知的罗杰斯和马斯洛。

### 逐页讲解

知识长尾是 LLM 幻觉最顽固的模型侧来源之一。一个专有名词在训练数据中的出现频率服从幂律分布（Zipf 定律）——极少数头部名词出现极其频繁（爱因斯坦、牛顿、乔布斯——每个人阅读量都极高），而绝大多数专业人物、冷门事件、小众概念只出现在极少数的专业文本中（长尾部）。模型对长尾部知识的记忆密度天然不足。

G.W.奥尔波特在人格心理学教科书和专业学术文献中是核心人物，但在互联网大众文本中出现的频率远低于心理学自媒体推文常提及的热门人物（罗杰斯、马斯洛、弗洛依德、荣格）。因此当模型被问到一个和奥尔波特相关的问题时，如果模型无法从内部参数中提取足够强的「记忆信号」，它可能会把问题映射到最相似的但又更常见的热门人物上（「谁谁是什么家」→ 大概率是某个常见心理学家的名字而非奥尔波特）。

RAG 对长尾问题的弥补非常直截——在专门的学术数据库中检索关于奥尔波特的文档，将检索结果和用户的提问一起作为上下文送进模型，模型就能看到「奥尔波特是美国心理学家，提出了人格特质理论」等精确信息，从而避免「猜测」到错误答案。


## 第 9 页 · 幻觉案例——曝光偏差（模型侧）

**PPT 内容**：
- 曝光偏差：由于模型训练（Teacher Forcing——喂入正确上文）与推理（自回归——喂入自己生成的带有误差的上文）存在差异，导致模型在实际推理时输出的概率分布偏离预期。
- 图解——Ground Truth 训练分布（正确的连贯上文）vs 推理时的自推上文（可能包含累积误差）。模型从未学习过从带误差的上文状态下恢复，因此在长生成中可能出现逐步的内容「漂移」。

### 逐页讲解

曝光偏差是 Teacher Forcing 训练范式的固有副作用。训练时，模型在每一步看到的都是正确的人类撰写文本作为上下文——即使模型自己的预测有错，损失计算后下一步仍然用真实上文作为输入。但推理时——模型喂入的是自己上一步输出的 Token——如果上一步出现了微小偏差（一个语气上的偏移、或一个事实的小错误），下一步将在这个偏差的基础上继续预测——偏差叠加偏差——一段时间后偏离了训练分布中的任何典型状态——模型在这种「陌生地带」行为不可预知——可能生成逻辑混乱甚至完全脱离主题的内容（幻觉的放大器）。

RAG 提供的外部文档可以作为「上下文纠正源」——检索到的正确事实文档在 Prompt 中占据很大比例和注意力权重——为模型提供了一个稳固的语义锚点——即使模型自推开始发生小幅漂移——外围的忠实文档上下文会往回「拉」住概率——减少漂移对生成最终正确性的影响。


## 第 10 页 · 幻觉案例——解码偏差（模型侧）

**PPT 内容**：
- 解码偏差：模型解码策略中的随机因素可能影响输出的准确性。
- 配图分为两半——符合常理的输出（采样到高概率 Token，输出流畅合理）和不符合常理的输出（偶尔采样到了低概率的不合理 Token，随机误差导致整体失实）。

### 逐页讲解

解码策略（Top-K / Top-P / Temperature 采样）为生成引入了随机性——目的是保证多样性和创意性。但这把双刃剑的另一面是——一定的低概率空间中的 Token 也可能偶然被采样到——而这个不合理 Token 一旦进入自推的上文——将连带影响后续所有预测分布发生偏移——可能产生内容失实的下游影响。

比如在高温度采样下——一句本来应该是「爱因斯坦出生于德国乌尔姆」的事实——随机采样偏差可能导致「爱因斯坦出生于德国汉堡」——即使这个错误对同类其他案例中概率很低。幻觉由此生。

RAG 在这里的作用是「提拉正向概率」——因为检索文档中明确记载了正确答案（如「出生于乌尔姆」），这些 Token 在 Prompt 上下文中被前置激活加权——随机采样朝向正确 Token 的概率被显著提升——降低偏离风险。


## 第 11 页 · RAG 止「幻」——开篇宣言

**PPT 内容**：
- 如何缓解幻觉问题呢？类比人类的解决方式——当我们遇到无法回答的问题时，通常会借助搜索引擎或查阅书籍资料来寻找相关信息。
- 图解——2023 年的考拉数量大约有多少？→ 大模型没有这个最新数据，无法正确回答。→ 模型先检索到外部文档（2023 年调整后的考拉种群估计数在 86,000 至 176,000 之间）→ 然后基于检索到的数据生成准确答案。

### 逐页讲解

⭐ 这一页的道出的是 RAG 的根本隐喻——**让 LLM 像人类一样「查资料再回答」**。

当一个人被问到自己不确定的问题时——本能的反应是什么？查手机、问搜索引擎、翻参考书。几乎没有人会闭目沉思然后将脑中零散记忆强行拼凑成一个「猜测」来回答——但在 RAG 出现之前，LLM 就是这样工作的（闭着眼睛从它的参数记忆中直接出概率分布——不留额外参考信息）。RAG 赋予了 LLM 「先查资料」的机制——在回答前先把外挂知识库中与提问最相关的文档检搜进来——将这些外部知识作为 Prompt 的一部分和原问题一起交给 LLM——最终 LLM 基于外部文档提供的事实依据来生成精确答案——大幅减少对自身脉冲记忆（且可能错误）的依赖。


## 第 12 页 · RAG 的核心思想

**PPT 内容**：
- 因此，对于大语言模型，同样可以通过检索与问题相关的信息进行辅助，从而有效缓解「幻觉」现象，大幅提升模型的生成质量。这便是检索增强生成（RAG）的核心思想。
- 图解更详细的流程：用户提问 → 知识检索模块从外部知识库中找到相关文档 → 这些文档与原问题一起输入 LLM → 生成基于文档事实的答案。

### 逐页讲解

RAG 的核心公式可以简化为：

**用户问题 + 检索到的权威文档 = LLM 的完整输入 → LLM 基于文档事实生成答案**

这其中的关键转变在于——LLM 的生成不再主要依赖它内部参数记忆的统计模式（这些模式会因上述七类原因产生幻觉），而是「临场」从外部知识库中提取当前问题对应的精准信息作为事实锚，然后在这一锚点基础上组织语言表述。

与第十章的「自回归生成」对比——传统自回归 LLM：P(答案|问题, 模型参数θ) ——仅依赖内部θ。RAG：第一步：检索最相关文档 D = Retriever(问题, 外部知识库)；第二步：P(答案|问题, D, 模型参数θ) ——受外部文档信息增强约束。这一步之差从根本上改变了模型行为的可靠性来源——从隐性记忆变成显性证据。

补充——文档信息用于增强 LLM 的最常用方式是 Prompt 拼接——直接把检索到的文档文本放在用户问题之前或之后，拼接进同一个上下文窗口。


## 第 13 页 · RAG 的概念与两大组成模块

**PPT 内容**：
- RAG（Retrieval-Augmented Generation），即检索增强生成，是一种从外部数据库中检索相关信息来辅助改善大语言模型生成质量的系统。
- 一个基本的 RAG 框架主要包含**知识检索**和**生成增强**两大模块。
- 图解：知识检索模块负责对输入问题进行编码并从知识库中快速检索出相关文档。生成增强模块接收检索到的文档和原问题，利用大语言模型生成最终的答案。

### 逐页讲解

这一页将 RAG 解构为两个核心功能单元：

**知识检索模块**（Retrieval Module）：接收用户原始问题 → 进行编码（可能是关键词提取/查询改写/查询增强）→ 对外部知识库进行检索 → 返回若干最相关的文档片段。检索方式可以是基于关键词匹配的传统稀疏方法（如 TF-IDF），也可以是基于语义向量的稠密方法（将查询和文档同时编码为高维向量后在公共空间算余弦相似度检索Top-K）。

**生成增强模块**（Generation Module）：将知识检索模块返回的相关文档 + 用户的原始问题组合成 LLM 的完整 Prompt → 经过 LLM 的自回归生成得到最终答案。

这种模块化设计使得 RAG 容易迭代和升级——可以独立替换「更强的检索器」或「更好的生成策略」而不用改动另一个模块。


## 第 14 页 · 知识检索模块——两种主流检索算法

**PPT 内容**：
- 知识检索模块的具体运作方式——对输入问题进行编码 → 从大规模知识库（如维基百科）中高效检索出与问题相关的文档。
- 常用的检索算法包括——基于关键词匹配的**稀疏检索算法**与基于神经网络的**稠密检索算法**。

### 逐页讲解

两种检索算法的对比如下：

**稀疏检索算法**（基于关键词匹配）：用 TF-IDF、BM25 等经典信息检索方法，把用户问题和知识库文档都表示成稀疏词频向量，通过计算向量内积来度量问题与文档的相关性。优点是计算快、基于确切的词匹配，不需神经网络训练；缺点是纯字面匹配——同义改写（「汽车」vs「轿车」）可能匹配不到——导致词汇语义缺口现象。

**稠密检索算法**（基于神经网络）：用预训练语言模型（如 BERT 编码器）把用户问题和知识库文档各自编码成稠密向量（~768 维），再通过两个向量之间的余弦相似度 / 内积来衡量语义相关性。优点——「汽车」和「轿车」在稠密向量空间中的距离很近——语义关联可被自动捕捉；缺点——需要大量计算预热（预先把所有知识库文档编码为向量并存储在大规模向量数据库）。

现代 RAG 系统往往采用「混合检索」——先用稀疏检索快速筛选一批候选，再用稠密检索对候选做精细重排序（重排见后文）——融合两者优势。


## 第 15 页 · 生成增强模块——Prompt 拼接为主流

**PPT 内容**：
- 生成增强模块将知识检索返回的文档和用户原始输入进行整合，作为大语言模型的新输入。
- 利用检索文档和输入问题 → 生成最终的输出序列（由如 LLAMA 等预训练语言模型完成）。生成器利用外部知识的最常见方式——通过 Prompt 的形式——直接把检索文档文本拼接到问题前面作为上下文。

### 逐页讲解

目前在工业界和学术界——Prompt 拼接（将检索文本前置到用户问题之前形成完整上下文）是所有 RAG 系统增强生成的主流和默认方式。一个大致的 Prompt 模版如下：

```
请基于以下参考资料回答用户问题。如果参考资料中没有相关信息，请如实说明。

参考资料：
[检索到的文档1]
[检索到的文档2]
...

用户问题：xxxxx？
```

这种方式的优势是「零成本」——不需要修改 LLM 的任何内部结构，不需要对其参数做任何微调或调整，只需在调用 API 时将外部的文本拼在 prompt 的前面部分即可立刻生效。这意味着对于闭源的、无法修改模型权重的黑盒模型（如 GPT-4、Claude），Prompt 拼接是唯一可行的外部增强方式。同时对于开源可微调的模型——Prompt 拼接也是最简单且初始性价比相对更高的方案。

但纯粹的 Prompt 拼接也有瓶颈：如果检索到的文档太长（可能几千 token）——拼接后大幅度增加推理成本（自注意力计算量与序列长度的平方成正比）以及可能超出模型的上下文窗口限制。

## 第 17 页 · RAG 架构分类——黑盒 vs 白盒

**PPT 内容**：
- 在 RAG 中，大语言模型根据参数是否可感知和可调节，可分为「黑盒」模型与「白盒」模型。
- 常见闭源模型（视为黑盒——参数不可访问、不可修改，仅可通过 API 调用）；开源模型根据是否对参数进行微调，既可视作「白盒」（可修改参数），也可保持为「黑盒」（仅调用不做微调）。

### 逐页讲解

这是 RAG 架构分类最基础的维度——模型参数的可访问性。

**黑盒模型**：模型的权重、隐藏状态对用户完全不可见——只能通过 API 输入文本 → 获取输出的 logits / 文本。如 GPT-4、Claude、Gemini 等商业闭源模型。对其进行 RAG 的方式只能是在输入端做文章——把检索到的文档作为额外 Prompt 上下文拼进去——无法改动模型内部的参数以更直接地适应 RAG 检索的信息。

**白盒模型**：模型权重完全可见、可修改。如 LLaMA 系列、Qwen 系列、DeepSeek 系列等开源模型。对其做 RAG，不仅可以在输入端拼接文档（和黑盒同策略），还可以对模型参数进行针对性的微调——如检索器和 LLM 的协同微调——使得 LLM 内部的语义表示也更适应将检索外部知识作为信号源并信任地使用（而非过度依赖内部记忆幻觉）。

同一个开源模型若仅使用 Prompt 拼接而不对参数做改动——等效为黑盒（虽然它在技术上本可被修改）。因此并非只有闭源才是黑盒——用户在不做微调时同样利用的是黑盒的增强方式。


## 第 18 页 · RAG 两大架构分类矩阵

**PPT 内容**：
- 从是否对大语言模型的参数进行更新的角度出发，RAG 架构可分为两大类：
- 黑盒增强架构（Black-Box Augmentation）——语言模型参数不更新
- 白盒增强架构（White-Box Augmentation）——语言模型参数参与更新

### 逐页讲解

以参数更新为分界线，两大类各自下属又按检索器是否微调分出更细的子类。下面将逐渐展开每种架构的细节。

黑盒增强——模型的参数始终冻结不参与改变。优点是不需要算力资源去跑训练脚本、不需要 GPU 更新权重，任何 API 调用者都可以轻松使用。缺点是模型本身的知识和检索文档之间的交互比较表面——仅通过 Prompt 拼接交叉，未能从参数层更深度地进行适配。

白盒增强——对语言模型或检索器的参数做训练/微调——提升它们之间的语义协作深度的同时也引入了训练开销和显存压力——但能产出比黑盒更优的「深度生成增强」。


## 第 19 页 · 黑盒增强架构的两个子类

**PPT 内容**：
- 黑盒增强架构可根据是否对检索器进行微调分为两类：
- 无微调架构——检索器和大语言模型均不进行任何微调，仅依靠它们在预训练阶段掌握的能力完成检索和生成任务。
- 检索器微调架构——语言模型参数不变，但检索器根据语言模型的输出反馈进行参数的针对性调整。

### 逐页讲解

**无微调架构**（最为普遍、最易实现）：检索器和 LLM 「原样搬运」各自的能力分头完成——检索器用预训练/预设的规则（如 TF-IDF 或 BERT 编码器）去检索外部知识库的 Top-K 文档；LLM 原样接收拼接后的 Prompt。两个组件互不干扰、维护成本最低。这种模式广泛用于快速部署原型或资源受限的场景。代表方法是 In-Context RALM——直接做 Prompt 拼接作为上下文。

**检索器微调架构**：由于黑盒 LLM 的参数不能修改，但我们仍然可以修改「为它提供文档的检索器」——根据来自 LLM 的 `信号` 来调整检索器的排序偏好，使得检索出来的文档更有可能帮助该 LLM 生成更好答案。方法如 REPLUG——用 LLM 输出的困惑度（PPL）来反哺监督检索器——PPL 越低意味着检索文档与 LLM 自身知识不冲突且对生成帮助越大→以此反馈信号微调检索器使其偏向检索低 PPL 文档。


## 第 20 页 · 白盒增强架构的两个子类

**PPT 内容**：
- 白盒增强架构也可根据是否对检索器进行微调分为两类：
- 仅微调大语言模型——检索器参数保持不变，语言模型根据检索器提供的信息进行参数调整。
- 检索器与大语言模型协同微调——检索器和语言模型迭代交互、一起协同微调。

### 逐页讲解

**仅微调大语言模型**（检索器不动）：保留现成的检索器（如原样使用的多模稠密检索模型或稀疏 BM25），只把 LLM 的参数在下游任务上微调（让 LLM 学会「怎么顺着检索回的外部文档生成正确答案」）→ 代表方法是 RETRO——在语言模型的中间层通过交叉注意力把检索文档的向量直接注入模型的隐藏状态中——而不是仅在 Prompt 表层的拼接。

**协同微调**（检索器和 LLM 同步更新）：检索器负责选出哪些文档应该被强化地融入 LLM 的生成；LLM 根据这些文档的优劣调整参数；与此同时检索器也得到反馈相应的梯度来改进自己——两者在训练中同时提升、互为提升的信号提供者。代表方法 ATLAS——以 KL 散度作为联合损失函数——同步优化检索器和 LLM 使得检索器输出的文档分布与之在 LLM 生成时实际的文档贡献分布趋于一致。这本质上是让检索器和生成器彼此不停匹配至「天然嵌合」的状态——协同微调达到了最高的 RAG 性能但也消耗最大的计算资源。


## 第 21 页 · 黑盒增强——无微调架构（In-Context RALM）

**PPT 内容**：
- 检索器和大语言模型在 RAG 过程中参数不更新，直接组合使用来完成生成任务。
- 代表性方法——**In-Context RALM**。其直接将检索器检索到的文档前置到输入问题前作为上下文。
- 参考文献：Ori Ram et al., 2023. In-context retrieval-augmented language models. TACL.

### 逐页讲解

In-Context RALM 是 RAG 最简单的基线形态——检索器用现成的预训练编码器（例如 Contriever 或 BERT Embedding）把用户的查询编码→和预置的知识库中所有文档的向量进行相似度排序得到 Top-K 文档→把这几段文档全文拼到用户原问题之前形成一个长 Prompt→喂给 LLM 回答。

在这种模式下搜索器和 LLM 均不作任何微调或适配——完全依赖 LLM 自身的指令跟随（in-context learning）能力来「看懂」检索来的文档内容并依之生成正确的答案。这个方法的好处是所有模型都可以直接用——不修改权重、不需要额外训练数据——成本最低、部署最方便。缺点是两个组件之间的磨合不够深——检索器可能返回与 LLM 生成偏向不特别吻合的文档，LLM 也可能偶尔无视或误读检索到的文档（仍去依赖它自己的内部偏差记忆）。实验表明 In-Context RALM 相比完全不检索的原始 LLM 有显著提升（困惑度明显更低）但仍留有大量改进空间。


## 第 22 页 · In-Context RALM 方法的优缺点

**PPT 内容**：
- In-Context RALM 方法与 LLM 解耦，易于实现，计算成本最低，相比于无检索能提升性能。
- 然而，该方法中 LLM 与检索器缺乏交互——完全依赖 LLM 的指令跟随能力。RAG 的整体效果难以得到保证。

### 逐页讲解

这一页给出了无微调架构最本质的两面性：

**优点面**：和任何 LLM 的组合都零成本——不需要重新训练 LLM、不需要训练检索器。你只需挂一个向量数据库和一个内置的检索逻辑（几行代码可完成）。相比于只让 LLM 不检索直接从参数记忆中生成——In-Context RALM 的额外文档确实帮助了事实性问题的改善。

**缺点面**：检索器和 LLM 完全是「陌生人配合」——检索器按照自己的语义模型去排序哪些文档最相关；而 LLM 对文档的「可用性评估」和检索器的评估并不完全一致。有时检索出的文档涉及冗余、过时、偏颇信息——LLM 不一定能够有意识地自主规避——完全依赖它的通用指令解读能力是否能够「判断」哪些部分是可信可用的——没有经过专门的针对性的训练或适配来做到这一点。

这就是后续章节中检索器微调、LLM 微调等更深度增强出现的原因。


## 第 23 页 · 黑盒增强——检索器微调（REPLUG）

**PPT 内容**：
- 大语言模型参数固定，检索器参数根据大语言模型的输出进行更新，使检索器能更好地适应大语言模型的需求。
- 代表性方法——**REPLUG**。其使用语言模型的困惑度作为监督信号来训练检索器——以检索出能够显著降低语言模型困惑度的文档为目标。
- 参考文献：Weijia Shi et al., 2023. Replug: Retrieval-augmented black-box language models.

### 逐页讲解

REPLUG 试图解决「检索结果不完全被 LLM 适应」的问题——虽然 LLM 参数无法改变，但可以通过 LLM 对检索文档「喜欢还是不喜欢」的定量信号（困惑度 PPL）反哺给检索器的训练使其产生更受欢迎的检索结果。

PPL = 语言模型对一段文本的困惑度——困惑度越低→模型越「理解、不感到意外」这段文本——该文本的统计规律在模型的知识空间吻合度高。如果对某个检索文档——当它和原始问题拼接后——LLM 对它的 PPL 显著降低（说明其内容对 LLM 生成正确答案非常有帮助——降低了不确定性），那么这个文档就可以作为检索器训练的「正反馈样本」——要更多地朝这类文档的属性优化检索器的参数——以便未来遇到同类型问题时能优先返回该类高价值文档。

这是在不触碰不可训练的黑盒模型条件下优化 RAG 整体链路性能的最经济有效路径之一。实验表明 REPLUG 微调后的检索器能为后续的 In-Context 生成提供更高质量、更贴合该 LLM 统计学结构的对外文档。


## 第 24 页 · REPLUG 方法的优缺点

**PPT 内容**：
- REPLUG 更新检索器以迎合 LLM 的需求，无需更新 LLM，成本低，效果良好。
- 但是 LLM 参数固定——可能无法与检索器良好适配（因为检索器和 LLM 始终是不可相互完全同步的二分系统）。

### 逐页讲解

REPLUG 是一种「中间路线」——投入少量计算调整检索器——使其偏好的文档在下游更好地服务于固定不变的 LLM——性能提升可观。但它仍是黑盒增强——LLM 对检索反馈信号的生成严格被限于固定参数空间内——两个系统在语义映射上不可避免地仍存在一定的 gap——完全的端到端协同优化无法在这个框架内部解决。要实现检索-生成的深度适配——最终还得转向白盒架构（后文）。


## 第 25 页 · 白盒增强——仅微调语言模型（RETRO）

**PPT 内容**：
- 检索器参数保持不变，大语言模型根据检索器输出对自身参数进行更新。
- 代表性方法——**RETRO**。其通过交叉编码，将检索信息动态地融合到大语言模型的隐藏状态中。
- 参考文献：Sebastian Borgeaud et al., 2022. Improving language models by retrieving from trillions of tokens. ICML.

### 逐页讲解

RETRO（Retrieval-Enhanced TRansfOrmer）在白盒增强方向做出了一个突破性创新——**不只是在 Prompt 表层拼接检索文本，而是通过交叉注意力把检索到的文档的向量表示直接注入 LLM 的每一层隐藏状态中**。

具体架构——RETRO 在 LLM 的多个中间层加入交叉注意力模块（类似 Encoder-Decoder 架构中的交叉注意力设计）——这个模块的 Query 来自当前层产生的隐藏状态（LLM 内部）——而 Key 和 Value 由外部的检索器返回的邻居文档的 BERT 编码向量提供。这样检索到的信息可以「在模型内部」在多层级影响语义生成过程——不仅仅是 Prompt 层面的输入。训练时——固定检索器——但让 LLM 的交叉注意力参数和其他参数一起训练——学习如何最有效利用邻居文档提供的事实信息来降低预测下一词的困惑度。

由于这种增强整合得「更深」，它拥有比 In-Context RALM 更稳固的信息依赖基础——即便检索文档很长——也无需完全占据模型的有限上下文窗口——因为它通过更紧凑的、学习到的交叉注意力方式融入而非 expand 为长串 Token。


## 第 26 页 · RETRO 方法的优缺点

**PPT 内容**：
- 该方法能优化语言模型生成能力，使其能够更好地利用检索到的外部信息。
- 但微调资源需求高——需要在大量的训练数据和 GPU 上训练交叉注意力适配——且微调效果依赖于原生检索器的性能（检索器不参与调整，若检索结果一开始就不相关——LLM 也难以从中获益）。

### 逐页讲解

RETRO 在生成质量上有显著提升（相比纯 Prompt 拼接）——但也具有高投入成本——需要对 LLM 做一次全量（或部分量大）的微调并设计交叉注意力辅助结构。并且如果底层的检索器返回一些完全不相关的文档——即使 RETRO 的交叉注意力再强——提取出的有用信号也极少——这就是检索器瓶颈对全文统的限制。这个瓶颈在下一个方法——协同微调（ATLAS）中得到解决——检索器也同步学习以提供高质量相关文档。


## 第 27 页 · 白盒增强——协同微调（ATLAS）

**PPT 内容**：
- 在检索器和语言模型协同微调架构中，检索器和语言模型的参数更新同步进行。
- 代表性方法——**ATLAS**。其使用 KL 散度损失函数来联合训练检索器和语言模型——以确保检索器输出的文档相关性分布与文档对语言模型的贡献分布相一致。
- 参考文献：Gautier Izacard et al., 2023. ATLAS: Few-shot learning with retrieval augmented language models. JMLR.

### 逐页讲解

⭐ ATLAS 是目前 RAG 技术栈中集成度最高的白盒方案——检索器和 LLM 完全联合训练——形成一个自我对齐的闭环。

核心机制——检索器产生一个文档相关性分布 P_retr(doc|query)（在所有可检索文档上的概率分配）；LLM 在生成答案时产生一个文档的隐性「贡献分布」（分析 LLM 在生成各个正确 Token 时对各文档的关注程度或信息贡献，它是训练中通过梯度比对解析出来的分布）。ATLAS 用 KL 散度作为损失来推动这两个分布一致——检索器会更倾力对着那些实际上在生成过程中被 LLM 所重点调用的文档做概率分配——而 LLM 也同步学习更合理地利用检索到的文档。

训练后——检索器和 LLM 是「天然的拍档」——检索器查出来的文档是 LLM 最擅长理解和最需要的类型——LLM 对检索器返回的文档类型天然信任和善于利用。协同微调在多项问答和多语言任务上取得了远超其他所有基本 RAG 部署方式的性能。代价——需要巨额的计算资源来对两个大网络同时做反向传播——且训练过程实现的工程复杂度更高。


## 第 28 页 · ATLAS 方法的优缺点

**PPT 内容**：
- 检索器和语言模型能够在训练过程中深度交互，不断优化 RAG 性能。
- 然而，资源需求高；且实现与训练过程复杂。

### 逐页讲解

ATLAS 处于 RAG 性能最高梯队的位置——但训练复杂度和成本也最高。需要同时维护双网络的优化器和梯度调度——对训练基础设施要求高——通常只有资源充裕的头部团队才能对最大的 LLM 和检索器做完整的协同训练。在中小规模场景中——纯 Prompt 拼接（In-Context RALM）或仅微调检索器（REPLUG）仍然是性价比较高、容易部署的过渡方案。


## 第 29 页 · 四类 RAG 架构对比总结

**PPT 内容**：
- 对比表格总结四种架构的优缺点——

| 架构分类 | 代表方法 | 优点 | 缺点 |
|---------|---------|------|------|
| 黑盒增强-无微调 | In-Context RALM | 与LLM解耦，易实现，成本最低 | LLM与检索器缺乏交互，效果难保证 |
| 黑盒增强-检索器微调 | REPLUG, AAR | 更新检索器迎合LLM，成本低效果良 | LLM参数固定，可能无法与检索器良好适配 |
| 白盒增强-仅微调LLM | REALM, Self-RAG, RETRO | 优化生成能力，更好地利用外部信息 | 计算成本高，微调效果依赖检索器性能 |
| 白盒增强-协同微调 | ATLAS | 检索器与LLM深度交互，不断优化性能 | 成本高，实现与训练过程复杂 |

### 逐页讲解

⭐ 这张对比表是 RAG 架构板块的核心总结——它提供了一个完整的决策参考框架——根据预算和性能要求选择合适的架构层级。一般来说——入门级最先试 In-Context RALM（即插即用、无训练成本）；初步优化用 REPLUG 训练检索器适配当前 LLM；深度优化上白盒（RETRO/ATLAS）。从学术观点来看——追求 RAG 最佳效果毫无疑问是协同微调；从企业实际实施——多数停留在黑盒增强阶段（低风险、易部署、安全可控）。


## 第 30 页 · 板块过渡——知识检索

**PPT 内容**：
- 过渡页——02 知识检索

### 逐页讲解

进入 RAG 的第二大板块——知识检索。这个板块将深入拆解从前端的「查询到知识库」完整链路——包括知识库构建（数据采集、清洗、分块、增强）、查询构建（同义改写、多视角分解、背景生成）、文本检索器（稀疏、双编码器、交叉编码器、生成式、图检索）、最后的检索结果重排和索引效率优化。


## 第 31 页 · 知识检索的总览

**PPT 内容**：
- 知识检索——如何检索出相关信息来辅助改善大语言模型生成质量的系统。

### 逐页讲解

知识检索模块是 RAG 的信息入口——其输出质量直接决定了后续生成增强的信息基础。被称作「rubbish in, rubbish out」的原则同样适用——如果检索器返回的是低质量或不相关的文档——LLM 再有强大也无奈——且还可能被不相关文档误导而产生新的错误。因此 RAG 系统的核心工程瓶颈常常不是 LLM 本身——而是检索质量的提升。


## 第 32 页 · 知识检索四阶段

**PPT 内容**：
- 知识检索通常包括四个部分：知识库构建、查询构建、文本检索、检索结果重排。

### 逐页讲解

知识检索可分解为一条流水线的四个串行环节：知识库构建（准备可检索的数据集和索引）→ 查询构建（接收用户原始问题，做语义增强和内容扩展）→ 文本检索（用各种检索器从知识库找出 Top-K 相关文档）→ 检索结果重排（用更强的交叉编码器对初筛结果做精细打分重排，提高结果精确度）。后续将逐个详细展开每个阶段的技术原理和方法。


## 第 33 页 · 知识库构建——数据采集与预处理

**PPT 内容**：
- 数据采集与预处理为构建知识库提供「原材料」。
- 数据采集——从不同渠道整合、转换多元数据资源，将其转换为统一的文档对象。例如维基百科语料库的采集内容不仅包括正文，还包括一系列元信息，如标题、目录、分类等。
- 图解：各种来源的文本文档（PDF、网页、数据库记录）被统一采集到一个存储库。

### 逐页讲解

知识库的「原材料」来自于各种异构数据源之间的整合——维基百科、企业内部知识 Web 平台、项目文档 PDF、代码库的注释、邮件记录、政策法规文本、医学资料库……这些不同格式、不同语言、不同质量的史料必须经过采集→格式转换→统一为一种索引系统能接受和处理的干净文档格式。重要的不仅是卷内文——各类元信息（目录标题、分类标签、来源时间戳、数据源权重）也要随一采集——这些在未来做检索过滤、以及时效权重加权的时候会起到关键作用。


## 第 34 页 · 数据预处理——清洗与分块

**PPT 内容**：
- 数据预处理为知识库的后续使用进行清洗和结构化。
- 数据清洗：清除文本中的干扰元素，如特殊字符、冗余空白、格式控制码等。
- 文本分块：将长文本切割成较小的单元——例如把一篇长文章分为多个短段落——每个段落就是一个检索基本单位。

### 逐页讲解

**数据清洗**——纯文本化过程中所产生的各种垃圾（多余的 HTML 标签残留、特殊 Unicode 控制字符、连续的空行等）都需要删除。对于某些 PDF 转文字文本——还可能需要处理离奇的换行截断和表格识别错误等固有偏差。

**文本分块**（chunking）：RAG 的检索基础单元不是一张完整的书或一篇上万字的文章——因为如果检索出一个极其冗长的长篇文档作为上下文——会极大浪费推理成本和上下文窗口（Token 预算）。需要将长篇文章切割成合理大小的「块」→ 使检索出的是一个高信息密度、中粒度大小的文本片段。分块方法见下一页面详解。


## 第 35 页 · 文本分块的两种方法

**PPT 内容**：
- 文本分块后得到的文本片段更加精细——能更准确地匹配用户的查询意图，提升检索结果的相关性。
- 文本分块方法包括：
- 固定大小分块——简单直观，按照预先设定的固定长度将文本划分为若干块。
- 基于内容的分块——根据待分块的内容，对文本进行更贴合内容逻辑的分块。包括句分割/递归分割（用句号和换行分割句子；借助 NLP 库如 NLTK、spaCy 等进行句分割）、以及直接利用 LLM 按照语义进行更智能的分块。

### 逐页讲解

**固定大小分块**：设定一个固定的 Token 数量（如 256 个 Token 为一块），从头到尾均匀切分。相邻块之间可能有部分重叠（overlap）防止语义单位被恰好切在交界处丢失跨块联系（通常 overlap 10-20%）。优点是简单快速；缺点——可能在句子的中间切开（一个词被一切两半、或一个完整段落被强行打断）。

**基于内容的分块**：利用句子边界、标题层级或语义段落断开来做更有逻辑的切割。各种 NLP 库（spaCy、NLTK）可做句分割（按句号分句）；LlamaIndex 等框架已经提供了递归分割（按章节标题、按句群、按句子层层分解）和基于 LLM 的语义分块（对文本进行预读理解其逻辑段落后在自然断点切割）。效果更佳——但需要更多的预处理时间和部分 LLM 调用开销。


## 第 36 页 · 知识库增强——伪查询生成与标题生成

**PPT 内容**：
- 知识库增强——通过改进和丰富知识库的内容和结构，为查询提供「抓手」。
- 伪查询生成：利用大语言模型生成与知识库中的文档内容紧密相关的伪查询——这些伪查询可以作为相关文档的「键」——供检索时与用户查询进行匹配。
- 标题生成：利用大语言模型为没有标题的文档生成合适的标题——可以在检索时更准确地对应用户提问进行定位。

### 逐页讲解

知识库增强是提升检索质量的强有效手段。它的核心理念是——提前为每个文档块生成多个「可能的问法」（伪查询）。例如，一篇关于「考拉栖息地遭受砍伐威胁」的文档块——LLM 可以自动生成——「考拉生活在什么地方？」「为什么考拉数量在减少？」「什么威胁着考拉的栖息地？」等多个潜在用户问法。这些伪查询和文档块一并编码为向量存放在索引中→这样一来当将来有用户提出任何和这些伪查询相近的真实提问时——检索模块能通过向量空间中的相近度迅速将相关文档定位出来——即使文档块的自身描述和用户提问在措辞上不完全一致。

标题生成同理——如果某个文档片段原本没有标题（如旧版 PDF 中的一段独立文本），用一个 LLM 来对它做一个自动标题提炼——生成几个关键字式标题叠加到索引——供检索时进行多维度信息覆盖。


## 第 37 页 · 查询构建——语义增强与内容增强

**PPT 内容**：
- 查询构建旨在通过查询增强方式，扩展和丰富用户查询的语义和内容——提高检索结果的准确性和全面性——「钩」出相应内容。
- 增强方式分为语义增强与内容增强两大类：
- 语义增强——同义改写和多视角分解等方法来扩展、丰富用户查询的语义
- 内容增强——背景文档生成等方法挖掘与查询主题相关的内容，提供更全面的知识支撑

### 逐页讲解

查询构建是在用户提问**送进搜索引擎之前**——对它进行「预处理增强」——提高搜索命中率。两个维度——语义增强负责拓宽「同一种意思的不同表达」的覆盖范围；内容增强负责拓宽「和这个问题相关的前后脉络信息」的覆盖面。两个方向分别针对不同的检索短板。


## 第 38 页 · 查询语义增强——同义改写

**PPT 内容**：
- 同义改写：通过将原始查询改写成相同语义下不同的表达方式——来解决用户查询单一的表达形式可能无法全面覆盖到知识库中多样化表达的知识的问题。

### 逐页讲解

用户提问常常是他自己某种特定措辞习惯下的自发问法——而知识库中的相关文档却可能用完全不同的词述策略描述同个话题。例如用户提问「为什么有些人一到秋天就掉很多头发？」，知识库中相关文档却可能以「季节性脱发的原因分析」为题——提问和文档之间的词汇重叠度极低——向量匹配也可能因词差异性大而错过。同义改写将原用户提问自动用多个相同语义但不同词汇的表达重新同义转换成若干等效提问——分别去检索——最终将多路寻得的文档结果合并交给生成增强模块。这极大增强了单一查询词串在多源文本风格下的语义鲁棒覆盖。


## 第 39 页 · 查询语义增强——多视角分解

**PPT 内容**：
- 多视角分解：采用分而治之的方法来处理复杂查询——将复杂查询分解为不同视角的子查询——以检索到查询相关的不同角度的信息——从而生成更为全面的答案。

### 逐页讲解

复杂问题通常无法在单一短查询中包裹周全地调度所有可能的答案维度的信息。例如「一个人每天需要多少睡眠时间？这如何随年龄变化？睡眠不足对身体有什么影响？」——这一个问题可从三个角度拆解——1)不同年龄段的推荐睡眠时长；2)睡眠时长的基础生理学；3)睡眠缺乏的危害。将这几个子查询分别去检索各自专门化相关的文档——最终合并所有不同视角的结果→交给 LLM 完成综合性、多维度源引用的答案——比拿着一个综合的粗查直接去找全文联效率更高、各方面信息的调用率更全。


## 第 40 页 · 查询内容增强——背景文档生成

**PPT 内容**：
- 背景文档生成：旨在利用大语言模型在原始查询的基础上——生成与查询内容相关的背景文档——从而丰富查询的广度与深度。
- 示例：原始查询可能非常简短（「国宝动物爱吃什么？」）→ 背景文档生成模块在查询输入检索系统前先让 LLM 自我推衍生成一个和此主题相关的详细背景描述——辅助检索器找到更贴切的文本内容。

### 逐页讲解

当用户查询过于精简或模糊时——在检索前的「预扩展」很有帮助——LLM 可用历史知识生成一段推测背景（「中国的国宝是熊猫、澳大利亚的国宝是考拉……熊猫爱吃竹子、考拉爱吃桉树叶……」），哪怕这个自推的背景信息也可能局部不完美——它的价值不在于最终答案——而在于作为一个更细粒度的检索用锚——能使检索到的外部文档更贴近问题域和更全面。背景文档由于包含自推的细节词汇——提升向量匹配的丰富度——从而帮助定位出原本检索可能遗漏的边缘相关文档。


## 第 41 页 · 文本检索总览

**PPT 内容**：
- 给定知识库和用户查询，文本检索旨在找到知识库中与用户查询相关的知识文本。
- 检索效率增强旨在解决检索时的性能瓶颈问题。
- 图解：用户查询 → 检索器 → 知识库 → 相关文档排序结果。

### 逐页讲解

文本检索是整个 RAG 流水线的核心匹配环节。从技术形态看——检索器实质上是一个**排序函数**：对知识库中的所有候选文档与该查询的相关性计算一个分数——将分数最高的前 K 个返回给下游。不同检索器在排序函数的设计原理、精度-速度trade-off、是否可训练等方面存在根本性区别——下面几页将对三大检索器类别（基于判别式/生成式/图）以及具体的子类分别展开。


## 第 42-43 页 · 检索质量和效率对 RAG 性能的决定性作用

**PPT 内容**：
- 检索的质量（召回率、精度、多样性）会直接影响大语言模型的生成质量。
- 检索的效率也是评估整个 RAG 系统性能的关键部分——将极大影响用户的使用体验（检索太慢——用户就要等不耐烦）。
- 不同检索索引（向量数据库 + 各种索引方法）的查询响应时间随文档数量增长变化的曲线。

### 逐页讲解

检索质量与最终 RAG 系统的答案正确率强相关——因为 LLM 的生成完全以检索到的文档为知识基础。如果召回率低（漏掉了知识库中某些珍贵的相关文档）→缺失的关键证据将导致答案残缺或错误。多样性低（查出的几篇文档全都来自同一来源同一观点）→导致回答单向偏颇。因此检索模块在所有 RAG 组件中占据了首当其冲的关键位置。

效率方面——当知识库规模达到数十亿篇文档/数十 TB 文本——任何单次检索必须能够在毫秒级完成——否则整体时长就会变成不可忍受的慢应答。向量索引（ANN——近似最近邻）以及倒排索引（inverted index）的优化是工程化的核心——不同索引算法（PQ 量化索引、HNSW 图索引、LSH 哈希索引等）决定了检索的延迟和召回率之间的物法平衡。


## 第 44 页 · 三类文本检索器——判别式 / 生成式 / 图检索

**PPT 内容**：
- 常见的文本检索器可以分为三类：判别式检索器、生成式检索器和图检索器。

### 逐页讲解

三种检索范式代表了三种完全不同的信息提取哲学——判別式检索器传统信息检索模式（输入查询+文档 → 概率函数输出相关性评分并过排序器选出Top-K文档）；生成式检索器将检索任务直接转化为语言模型生成任务——给模型查询并让它「生成」出相关的文档标识符（DocID）；图检索器将知识组织为带关系边三元组的图数据库——用图遍历和子图匹配的方式找到和查询节点相关联的三元组信息子图。三种各适于不同检索环境语义关系复杂度的场景。


## 第 45-48 页 · 判别式检索器——稀疏检索 TF-IDF

**PPT 内容**：判别式检索器通过对问题和文档进行特征向量提取并计算相关性分数来排序文档。稀疏检索器——使用稀疏表示方法匹配文本——TF-IDF 是典型代表。TF-IDF 基于词频（TF）和逆文档频率（IDF）衡量词的重要性→用此重要性对文本进行编码，然后对两个 TF-IDF 向量进行余弦相似或内积计算以评估查询和文档的相关性。

### 逐页讲解

TF-IDF = 词频 TF × 逆文档频率 IDF。TF = 某个词j在文档i中的出现次数 / 文档i的总词数 ——该词在本篇文档内有多「频繁出现」。IDF = log(总文档数 N / 包含词j的文档数 df_j) ——用来惩罚过于普通的词（在所有文档中都高频出现的词——如「的」「是」「the」「a」——对区分某篇具体文档特异性不强因而 IDF 压低）。

最后查询和文档的自算得分 = sum_i (TF-IDF(词i, 查询) × TF-IDF(词i, 文档)) ——通过各词权重求内积实现稀疏向量的快速文本匹配。

计算示例（PPT中的家人/集美/们例子）——三句话 TF/IDF 完整逐步推导最终各句的 TF-IDF 向量——利用向量内积发现「句1和句3在家人、们节点呈现相似性」的关联并对句子进行匹配排序。

TF-IDF 的优势是快速直观不需训练、低算力。缺点是纯字面匹配——同义异形词（「汽车」vs「轿车」——两个字的TF-IDF权重条目完全独立——没有共享任何交叉评分）——检索不到相关但措辞不同的文档而白费。


## 第 49-50 页 · 双编码器稠密检索与 ColBERT

**PPT 内容**：双编码器检索器——将查询和文档先各自通过独立的编码器（例如两个 BERT 编码器）生成各自的低维稠密向量，再对这两个向量做点积/余弦相似度计算作为相关性分数。允许离线预先计算存储所有文档的向量 —— 匹配时只需在线编码一次查询向量并与预存文档向量做一次快速矩阵乘距离求解——匹配效率很高。但缺陷——查询和文档在编码阶段分别独立编码没有交互——ColBERT通过查询词和文档Token级的细粒度交叉相似度改善这个问题——对查询的每个 token 嵌入独立检索文档并取对应最相似度最大化得分和。

### 逐页讲解

**双编码器架构的本质优势**——所有文档可离线预编码为向量并将向量索引存储在向量数据库中；在线仅需处理对单个查询做一次编码——将查询向量和库中所有文档向量计算距离（ANN近似最近邻高效搜索大量文本）——时间复杂度远低于需要对查询和每份候选文档做全transformer推理的交叉编码器。

**ColBERT 的「后期交互」**——即避免了对全部全句级别做繁重交叉编码，但提升了句子交互粒度到每个Token级的比较——使检索相关性高于纯双编码器且速度仍远快于全交互式交叉编码器——为很多标准开源RAG管道中检索+重排的首选过渡。


## 第 51 页 · 交叉编码器——全交互深度打分

**PPT 内容**：交叉编码器「端到端」给出查询与文档的相似度。将查询和文档拼接后直接输入到模型中，模型输出一个介于 0 到 1 之间的相似度值。模型结构简单→能实现查询和文档间的深度交互。但计算量大——不适合在大规模检索阶段使用（需要对每个候选文档都重新做一次 Transformer 推理前向），更适用在小规模的重排阶段。

### 逐页讲解

全交互式交叉编码器（如 BERT/RoBERTa 的 cross-encoder 版本）对每一个查询都会和每一篇文档做深度 Transformer 嵌套计算——精度非常高但延迟也高——无法覆盖数亿/十亿规模文档集的全局海选。因此多阶段检索策略——初筛（双编码器/稀疏检索快速找出 Top ~100 候选文档）→重排（交叉编码器对这 TOP100 做精排）构成黄金配对——在速度和精确度上兼得。


## 第 52 页 · 向量数据库与索引加速

**PPT 内容**：知识库海量文本的向量编码储存在向量数据库中。对所有向量逐一比对非常缓慢低效——高效的向量索引方法（量化/哈希/树/图）被开发以加速近似最近邻搜索。向量数据库如 Milvus、Pinecone、Qdrant、Chroma 等和对多种主流向量索引的适配。展示了不同索引方法的查询响应时间随向量量的变化曲线。

### 逐页讲解

向量数据库是 RAG 系统集成的必备中间件——它不同于传统的关系型数据库——专为高维稠密向量的近邻检索设计。各类索引各有特点——乘积量化 (PQ) 索引将高维向量切分成多个子向量分别量化→通过低比特码本压缩大幅降存储和加速距离估算；HNSW（分层可导航小世界图）索引通过构造多层图逐级跳跃搜索→查询速度极快；LSH（局部敏感哈希）将高维空间分桶至哈希桶→相似向量大概率落入同一桶→加速检索。

推荐组合——主索引用 HNSW 图索引（高召回+极速），在超大规模时可结合 PQ 压缩降低存储。


## 第 53-56 页 · 生成式检索器——直接生成文档 ID

**PPT 内容**：生成式检索器直接将知识库中的文档信息记忆在模型参数中。在接收到查询请求时，能够直接生成相关文档的标识符（即 Doc ID）来完成检索而不需要显示地做向量相似度比对。训练——两阶段联合训练——索引任务（模型学习文档内容和对应DocID之间的映射关系）+检索任务（模型学习为特定查询输出一个潜在相关的候选文档ID排名列表）。内存效率高——仅需存储文档标识符和模型参数，总占用极小；生成速度快——仅需生成轻量的文档ID而无需查找外部向量库。

### 逐页讲解

生成式检索是对传统「索引-比对-排序」检索模式的根本性颠覆——它把大规模文档检索任务交给一个语言模型通过参数记忆来完成——文档的 ID 就是 LLM 的自回归生成输出之一。训练阶段——对每个文档:模型学习将文档全文映射到其对应的ID（学习索引）。然后训练查询→生成候选ID排名的检索映射。整个外部数据库实质上「蒸馏进了模型的参数内部」而不依赖硬盘/存储索引——内存占用非常小（仅模型参数本身），且所有检索动作都退化为一次 LLM 的生成调用（对某些轻量 DocID 输出速度极快）。

局限——对每个文档都需要指定、训练生成对应的 DocID，对动态变化的知识库——新增文档需重新进行增量训练——不如外部向量库灵活。


## 第 57-60 页 · 图检索器——结构化知识

**PPT 内容**：图检索器的知识库是图数据库——含开放知识图谱和自建图。一般由<主体, 谓词, 客体>三元组构成——能捕捉实体间的显式概念关系并允许人类和机器对知识进行结构化的理解和推理。图检索在三类场景中优于扁平的文档检索——需捕捉结构化关系、需避免「lost in the middle」冗长的文段信息在长上下文中被遗忘、需在全局层面上进行逻辑关联性检索。流程——图索引构建 → 检索（子图匹配及召回）→ 结合生成模块。


## 第 61-63 页 · 检索结果重排——交叉编码精排与 RankGPT

**PPT 内容**：检索阶段为保证速度常牺牲部分性能——可能检索到质量较低的文档。重排的目標就是对初筛检索得到的段落进一步细粒度排序。两类——基于交叉编码的方法（对每个候选文档用深度双向交叉编码器计算精准相关性分→重排）和基于上下文学习的方法（如 RankGPT 用 LLM 根据精心设计的 Prompt 评估文档并对候选队列重新排序）。后者代表性方法 RankGPT——将在Prompt中列出所有候选段落，由LLM（如GPT-4）综合上下文逻辑给出排序指导。

### 逐页讲解

交叉编码重排虽精度高但重量级——常在初筛的~100个候选窗内运行。RankGPT 的方法更具启发——直接用大语言模型自身的强大的语言推理来替代人工设计相关度评估器做「逻辑审读+综合打分」——在较少几条候选间深度交叉推理能力强——适合最终精选几个最佳引用源。


## 第 64 页 · 板块过渡——生成增强

**PPT 内容**：进入第三板块——检索器得到相关信息后，需要将其传递给大语言模型以期增强模型的生成能力。如何优化生成增强过程？

### 逐页讲解

过渡页——标志着从「怎么搜」切换到「搜到后如何利用」


## 第 65-76 页 · 何时增强——判断是否真的需要检索

**PPT 内容**：确定何时需要检索增强以确保非必要不增强。判断的核心是判断 LLM 是否具有内部知识——对于内部知识足以回答的问题不做增强以避免引入噪声。两类观测法——外部观测（Prompt 直问、观察训练数据/伪数据流行度估计模型知识）vs 内部观测（训练线性探针在模型隐层检测已知/未知状态）。流行度阈值——可设定维基百科/谷歌某实体访问量阈值作为类别——主判断是否需要查外援。

### 逐页讲解

这一板块的一个重要思路——RAG 并非对所有提问都需要一律触发。模型自带的知识——对高频/常识化的提问完全可以自主准确回答——强行加一个检索反而可能多此一举——增加延迟且可能引入质量不佳的外来文档打乱模型原本正确的参考答案。因此一个智能的 RAG 系统需要在行动前先审视「我该不该出外检索」。

外部观测——直接反诘模型自身某问题你是否懂得（Prompt 直问→容易导致模型过度自信「都懂」）；通过检查某个实体/知识点在外部搜索平台的近期热度（流行度）——比对照判断它在训练数据中的常见度（热门知识大概率已知、冷知识大概率长尾记忆差→此时需 RAG）。

内部观测——探针直接诊断模型内部的激活模式——类似通过「检测欺骗的微表情」探测模型真正在有/无知识时的暗中信号变化，而不是问它自己。


## 第 77-82 页 · 何处增强——输入端/中间层/输出端

**PPT 内容**：三种策略各自优缺的对比与场景。输入端增强主流易实现但长文档耗费计算加负担；中间层增强如 RETRO 深度融合隐藏状态但不能用于黑盒、需要架构改造和设计调整；输出端增强为在LLM 初步生成后调用检索文本进行后矫正（事实性检查和修正最终输出）。


## 第 83-86 页 · 多次增强——复杂问题分解与模糊问题细化

**PPT 内容**：对于复杂多跳推理问题——将父问题分解成含前后序关系的链条子问题→然后级联多次检索→每次检索结果引导下一子问题的生成→最终合并多源信息得出最终答案。对于模糊歧义问题——将可能意图分支化多路径各自探索→最后评判最优路径给出主答案。


## 第 87-100 页 · 降本增效——去除冗余文本与复用计算

**PPT 内容**：检索的外部知识常包含大量文本增加模型推理计算成本——可通过去除冗余文本（Token 级/FIT-RAG 子文本级/全文本级压缩）→去掉与生成不相干的低信息 Token 段落——输入 Token 可减少约 50% 的同时保持回复准确率不降。另一侧——对中间 KV 计算结果缓存复用（KV-cache + RAGCache 多级缓存框架）→减少重复 Token 的前向推理计算→显著提升多轮检索生成吞吐。

### 逐页讲解

FIT-RAG 是本板块最突出的代表作之一——通过将检索到的长文档打细分成多个子文本段——每个段由一个打分器评估其对生成贡献的事实质量评分——选取得分最高的一小段组合——既能减少 Token 总数又确保剩余 sub-texts 是对答案最关键的支撑——在多项实验中展示了输入缩减 ~50%的 Token 而回答准确率反超全量基线。


## 第 101-113 页 · 应用落地

**PPT 内容**：多模态 RAG——扩展到图像/病历/语音等多源异构信息辅助多模态医疗诊断或视频检索。AI Agent 中的 RAG——在 Agent 记忆/计划/行动各模块中嵌入检索接口提供外部环境的信息收集和决策前的事实补充。企业落地——知识库问答 (36.2%) 和支持类聊天机器人 (28.9%) 在所有企业 AI 应用场景中居前二—— RAG 是这一切的核心技术供给。
